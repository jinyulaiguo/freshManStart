# Workspace Rules

## AI 导师教学规范

### 1. 每周学习启动前置规范
在开启每一周的学习（如 Week 2 等）之前，AI **必须首先进行宏观总览与框架拆解**，禁止直接进入每日知识点讲解。
*   **动作要求**：
    1. 先对本周最终实战项目的整体逻辑、业务流程进行全景讲解（推荐采用交互式 HTML 看板或直观的流程图展示）。
    2. 将本周每日知识点串联起来，形成知识链路图。
    3. 清晰标注出系统的每个组成模块**“具体做了什么”**以及**“解决了什么痛点问题”**。
*   **物理存储规范**：
    1. 每周的宏观全景总览看板 **必须命名为 `overview.html` 并物理存放在该周的根目录下**（例如：`weekly/w09_memory_system/overview.html`），严禁放入具体的单日练习 `dayYY` 目录中。
    2. **启动脚本限制**：启动脚本（如 `start.sh`）与配套的 Web 调试 Dashboard 仅允许在每周的综合实战大项目（Project 阶段，如 Day 63）的单日目录中提供，用以物理拉起 Web 服务；在平时以理论与契约协议为主的 Daily 练习（如 Day 57 - Day 62）目录中，**严禁物理生成任何 `start.sh` 或单日 `dashboard.html`**，所有运行调试均必须完全收拢在 `practice.py` 内部的 `__main__` 调试主入口中。
*   **执行条件**：在完成上述宏观流程拆解、写入 `overview.html` 并获得用户确认通过后，方可进入具体的每日“六步教学”。

### 2. 每日知识点六步教学法与语言规范
对于本仓库中所有 Python/Agent 进阶知识点（如 Day 8 到 Day 14 等）的具体讲解、学习引导和代码实践，AI 必须严格遵守 [docs/six_step_learning_method.md](file:///Users/zhouyi/03.AI/03.freshManStart/docs/six_step_learning_method.md) 中定义的**“六步教学法”**，并强制执行以下讲解规范：
*   **自问自答模式**：讲解过程必须以问题为导向，通过“问（卡点与崩溃隐患）”与“答（技术解决方案）”的结构层层递进。
*   **工程语言规范**：禁止一切虚浮修饰词及文学类比（如“骨架”、“大门”、“血管”等），必须使用标准的软件工程与 Python 官方术语（如“类型契约”、“数据校验拦截”、“异常链传递”、“非阻塞并发”等）。

1. **第零步：定位并引导痛点场景** — 必须说明无此技术时的系统痛点/底层限制。
2. **第一步：建立直观类比与核心定义** — 使用常见类比，不超过三句话总结核心技术定义。
3. **第二步：提供最小但真实的代码示例** — 拒绝 Hello World，引导用户静态预测并 Debug。
4. **第三步：展示主动破坏与常见报错** — 主动提供防御性编程报错分析。
5. **第四步：设计默写与主动召回机制** — 提供核心逻辑的填空/默写题目。
6. **第五步：剖析真实开源项目的用法** — 分析优秀开源框架的真实场景、入参模型与异常处理。
7. **第六步：交付自底向上重构的最小引擎与测试** — 提供带单元测试的骨架以闭环验证。

## 练习文件与参考答案编写规范

### 3. 模版与标准答案分离规范
在构建或修改每日练习题时，必须严格遵守以下文件结构与内容分离规范：
*   **练习模版 (`practice.py`)**：此文件为学员练习专用。除基本的类定义声明、回调基类等骨架外，所有核心逻辑实现（如归约算法、反射机制、运行流程等）**必须仅包含 TODO 注释与 `raise NotImplementedError("TODO")` 或 `pass` 占位**，严禁夹带具体实现代码。
*   **参考标准答案 (`state_tracker.py` / `tool_runner.py` 等)**：此文件为参考标准答案。必须包含完整代码实现，且通过全部单元测试验证。

### 4. 编码前设计与详实代码注释规范
在修改或编写任何代码文件（无论是练习模版还是参考答案）时，必须遵守以下注释与设计规范：
1. **文件头部设计说明**：AI **必须首先在文件头部或核心类/函数的前方，以详细的 Docstring 或块注释形式写明该模块的设计方案（必须包含设计意图、类与函数结构、关键数据流流向、以及核心用例设计意图）**。在完成设计方案的文档化说明后，方可进行具体逻辑代码的编写。
   * **核心用例设计意图 (Test Case Design Intent)**：必须在头部显式阐明主入口测试用例的选取考量、真实业务场景意义，并列出具象化的核心验证点（例如：验证 Planner 的解耦识别能力、DAG 拓扑分层、变量占位符回填及并发性能加速指标等）。
2. **方法与步骤详实注解**：所有定义的类、方法和辅助函数，必须附带详尽的 PEP 257 标准 Docstrings，明确指出入参类型、返回值格式以及异常情况。同时，函数内部的核心控制流、分支判定与并发调度处，必须提供行内步骤块注释（Inline Comments），严禁出现无注释逻辑，确保代码 100% 可读与自解释。

### 5. 多方案对比物理隔离与冗余规范
当代码文件（包括练习模版和参考答案）中需要呈现多种不同阶段、不同技术演进或不同设计范式的实现进行对比时，**必须将它们在物理上彻底隔离在不同的代码板块中（例如使用清晰的板块分割线注释）**。
* **允许并鼓励有意识的冗余（Intentionally Redundant Design）**：每套方案应包含自己所需的完整依赖定义（如独立的类型契约、辅助合并函数等），哪怕这会产生代码重复。
* **严禁跨范式逻辑共享**：严禁多个对比方案共享同一个内部底层函数，以确保每一套对比代码在物理和逻辑上均 **100% 自包含**，从而彻底降低学习者的理解和认知负担。

### 6. 练习用例与运行入口规范
* **练习模版 (`practice.py`)**：不需要编写专门的单元测试文件（如 `test_practice.py`），但必须包含可以直接运行且带有友好 TODO 拦截提示的调试主入口（例如 `if __name__ == "__main__":` 搭配 `try...except NotImplementedError`），以方便学员在没有测试套件的情况下进行直观验证。
* **参考标准答案 (`async_basics.py` / `state_tracker.py` 等)**：必须包含可直接运行且带清晰控制台输出（stdout）的主入口，以直观演示其设计意图（如并发特征、超时限制或异常隔离）。



## 课堂笔记 (notes.md) 编写规范

### 7. 教学心智与静态文档解耦规范
* **六步学习法定位**：六步学习法是 AI 导师的“教学心智模型”与“动态交互框架”，旨在指导 AI 引导提问、设计练习骨架、组织默写填空等动态交互逻辑，**严禁将“第零步”、“第一步”等元步骤字面直接作为静态笔记（`notes.md`）的大纲或标题**。
* **笔记文档（`notes.md`）结构**：必须采用纯粹、自然的**软件工程技术大纲**进行模块划分（例如业务背景、协议原理、核心指标量化、异常防错设计等），并且在内容上潜移默化地融入痛点引导和概念定义。

### 8. 场景化教学与去冗余代码规范
* **业务场景关联**：在讲授抽象技术点时，必须代入一个**具体的、有代表性的 Agent 工业级业务场景**（如：多 Agent 并发代码审查系统），直观量化对比使用该技术前后的性能数据（如首字延迟 TTFT、总执行时间等），绝不空发议论。
* **严禁大面积源码**：静态笔记（`notes.md`）中**严禁贴入大面积可运行的 Python 源码**（源码细节必须收拢到 `practice.py` 与 `stream_engine.py` 中）。笔记中仅允许展示：
  1. **极简的核心逻辑伪代码**（不超过 20 行）。
  2. **直观的逻辑流程图 (Flowchart)**。优先使用流程图取代时序图以清晰表述代码内部的控制流决策路径。

### 9. Mermaid 流程图编写兼容性规范
* 在编写 Mermaid 流程图时，为了保证在各种 Markdown 渲染引擎下的最高兼容性，**所有包含特殊符号（如单/双引号 `'` `"`、问号 `?`、冒号 `:` 等）的节点文本必须一律使用双引号 `""` 进行包裹**。

### 10. 自底向上与“积木式”拼装架构规范
在构建每周实战项目或中大型 Agent 系统时，AI 必须严格践行自底向上（Bottom-Up）的工程拼装思想，禁止直接在主入口编写大而全的耦合业务逻辑：
* **微引擎物理隔离**：系统必须被拆解为数个高内聚、职责单一的“微引擎”或“适配层”组件（例如：独立的会话管理器、独立的上下文裁剪器、独立的故障降级控制器等）。每个组件作为一个独立模块封装。
* **单兵作战与单元测试**：拼装装配前，必须针对每个独立的“微引擎”编写相应的单元测试，确保各模块在边界条件、异常流和高并发下自身 100% 行为正确。
* **无逻辑控制主入口**：主入口程序（如 `main.py`）仅作为生命周期管理、事件循环以及各微引擎实例的“拼装积木墙”，只负责串联微引擎的数据流向 and 命令行 REPL 调度，不承担具体的算法实现和策略逻辑。

### 11. 网页设计与“温润知性极简主义”配色规范
所有生成的 HTML 页面、展示看板和前端组件必须严格遵循 [design.md](file:///Users/zhouyi/03.AI/03.freshManStart/.agents/design.md) 中定义的 **Warm Intellectual Minimalism (温润知性极简主义)** 设计风格。
* **配色系统**：
  * **Background (主背景)**：骨白 Bone-white (`#faf9f4` 或 `#faf9f4`)，严禁使用炫酷的暗色背景或高饱和度纯色。
  * **Surface (卡片/容器背景)**：纯白 Pure-white (`#ffffff`)。
  * **Primary / Text (主字色)**：深炭黑 Charcoal (`#141413`)，On-surface variant 为 `#464742`。
  * **Accent (点缀/CTA)**：杏粘土色 Clay accent (`#D97757`)，仅用于状态指示、关键强调及高优先级 CTA。
  * **Error / Success**：错误使用暖红 `#ba1a1a`（搭配浅红背景 `#ffdad6`）；成功使用有机绿 `#2e623a`（搭配浅绿背景 `#edf6ef`）。
* **边框与阴影 (Flat Aesthetics)**：
  * 严禁使用多重或强阴影，采用 **Tonal Layers** 和 **低对比度边框** 代替。卡片外围均使用 1px 细线边框（如 `#e9e8e3`），Hover 状态仅做细微边框变色或背景微调。
* **圆角比例 (Disciplined Rounded)**：
  * 避免过度饱满的圆角。小元素（按钮、输入框、标签）圆角大小使用 **0.25rem (4px)**；大容器（卡片、展示区）圆角使用 **0.5rem (8px)**。
* **排版系统 (Serif-Sans Contrast)**：
  * 标题类（h1, h2, h3, h4 等页面/区块标题）必须使用 **Source Serif 4** 衬线体，传递严肃而知性的人文质感；正文和 UI 操作层统一使用 **Inter** 无衬线体。

### 12. 真实 API 调用与配置管理规范
后续所有教学与练习案例中涉及大模型网络请求时，**必须统一使用真实的 API 接口调用进行功能演练与过关验证**，严禁使用 Mock 数据。
* **环境变量管理**：必须统一从项目根目录的 `.env` 配置文件读取 API credentials（如 `MINIMAX_API_KEY`、`MINIMAX_BASE_URL` 等）。
* **配置加载与网络工具提取**：为了避免在每日练习中重复书写非核心配置和网络底层交互逻辑，统一将配置加载器与基础 LLM 请求客户端封装在公共工具模块 `weekly/w04_prompt_and_http/utils.py` 中。在具体练习或标准答案中直接通过 `import` 导入使用，以保持核心功能代码的高内聚与低认知负担。

### 13. Git 提交信息（Commit Message）规范
所有代码库中的修改和练习提交必须严格遵循本项目的历史 Git Commit 规范，格式要求如下：
* **每日练习提交 (Daily Exercises)**：必须采用 `learn(wXX/dayYY):[中文说明]` 格式。
  * `wXX` 表示星期（如 `w04`），`dayYY` 表示天数（如 `day22`）。
  * 冒号 `:` 后面**紧跟中文说明，禁止加空格**。
  * 示例：`learn(w04/day22):CoT思维链、Few-shot样本库与Self-Consistency异步并行投票机制`。
* **每周大项目提交 (Weekly Project)**：必须采用 `project(wXX):[中文说明]` 格式。
  * 示例：`project(w02):生产级异步多工具调度引擎`。
* **其他工程配置提交 (Others)**：例如涉及工程基础环境配置变动，采用 `chore:[中文说明]` 格式。
  * 示例：`chore:更新忽略文件规则`。

### 14. 开源框架深度对比规范
若教学内容或用户要求涉及对主流 Agent 开源框架（如 LangGraph, OpenAI Agents SDK, OpenManus, smolagents 等）的深度对比或核心源码剖析时，必须通过以下规范沉淀为独立的技术剖析文件：
1. **多维对比矩阵**：首节必须提供直观的横向对比表格，覆盖设计哲学、控制流迁移机制、状态容器设计、工具调度方式、安全沙箱边界与核心适用场景等软件工程维度。
2. **物理流转图谱**：必须为每个涉及的框架绘制专属的 Mermaid 流程图，且严格遵守“节点特殊符号双引号包裹”的兼容性规范。流程图须展现出状态流转方向、条件路由分支与回环反馈闭环。
3. **核心控制与调度本质剖析**：严禁空洞泛化的描述，必须精准剖析各框架底层的实现机理（例如 LangGraph 的 Pregel 超步模型、OpenAI Agents SDK 的事件流挂起、OpenManus 的物理沙箱管理、smolagents 的受限 Python 解释器）。
4. **工程决策指导**：文档末尾必须提供基于真实工业级应用场景的技术架构选型决策指南。

### 15. Git 提交记录合并与修补规范
为了维护 Git 提交历史（Commit History）的整洁度与原子性，严禁针对单日练习或每周大项目生成零散、重复的多次提交记录：
*   **单模块原子性提交**：每日知识点练习（`dayYY`）或每周大项目（`wXX`）的所有代码改动、文档沉淀与测试代码，在最终交付时**必须且仅允许合并为一条唯一的 Commit 记录**。
*   **增量修复与代码重构修补**：在提交后如需进行 Bug 修复、代码重构、配置调整或文档补全，**绝对禁止生成新的 Commit 记录**。AI 必须在本地使用 `git commit -a --amend --no-edit` 命令进行“静默修补”，将修改就地融入已存在的该条提交中。
*   **远程仓库的同步规整**：如果上一次的提交已经推送（Push）到远程仓库，在进行本地修补（Amend）后，必须使用 `git push --force-with-lease origin [branch]` 命令进行安全性的强制覆盖推送，确保本地与远程保持 100% 同步且依然只有唯一一条干净的 Commit。
*   **禁止擅自提交铁律**：除非用户显式下达 Commit 或提交指令（例如明确说出“你可以提交了”、“进行提交/commit”），否则 AI **绝对禁止以任何理由自行在后台执行 `git commit` 操作**。所有修改必须保持在本地工作区（Unstaged/Staged 状态），静默等待用户的最终审查与提交指令。

### 16. 生产级架构师设计规范（Enterprise Architect Standard）
本仓库中所有每日练习（`practice.py`）、参考标准答案（`xxx_impl.py`）以及教学讲解与文档，**必须 100% 采用生产级、架构师级（Architect-Level）的工程标准与真实工业级业务场景进行设计**，严禁使用任何针对初学者的简化模型：
* **真实工业业务上下文**：严禁在代码中使用任何无业务含义的 `node_a`/`node_b`、`foo`/`bar` 或 `Hello World` 式占位代码。必须代入具体的工业级 Agent 业务场景（例如：多 Agent 自动化代码重构与 AST 安全审计引擎、高并发金融回测 Pipeline、分布式 Trace 审计链等）。
* **架构级容错与设计模式**：必须采用生产级的切面代理（AOP）、状态快照保险箱（State Snapshot Vault）、结构化诊断 Payload、多维熔断策略（如超步限额、运行耗时预算超时、状态指纹震荡检测）等架构模式，确保代码具备 100% 生产落地的参考价值。

### 17. 全局战略定位与双图谱映射规范 (Global Strategic Alignment Standard)
在进行任何单日（如 Day YY）的技术讲解、代码构建与交互引导时，AI **必须强制建立全局视角，从架构层面思考单点 API 的位置**：
1. **AI 导师认知锚定**：AI 必须参考 [docs/agent_learning_plan.md](file:///Users/zhouyi/03.AI/03.freshManStart/docs/agent_learning_plan.md) 与当周大纲（如 [docs/week_10_langgraph_core.md](file:///Users/zhouyi/03.AI/03.freshManStart/docs/week_10_langgraph_core.md)），在交互回答与提问引导中体现该日知识点在 26 周路线图中的战略意图与本周积木位置。
2. **课堂笔记 (`notes.md`) 解耦边界**：静态笔记 `notes.md` **必须保持纯粹的软件工程技术大纲**（聚焦于当天的业务痛点、底层原理、流程图谱、代码模式与性能指标），**严禁将全局规划说明、元规则描述或宏观路线图文本直接贴入 `notes.md` 中**，确保笔记纯粹高效、干货满载。

### 18. 内容自适应扩充与全局大纲同步规范 (Content Extension & Syllabus Sync Standard)
在进行单日（Day YY）知识点教学或代码构建时，若发现原有规划中的内容过于简单、模型过于初级或缺乏工业落地深度，AI **必须强制根据当周目标与全局 26 周路线图进行内容扩充与深度强化**：
1. **主动深化与告知**：AI 必须结合真实工业级业务复杂性，增加核心范式、防御机制、数据结构与恢复流转等内容，并在教学启动时**主动明确告知用户扩充的动机与增加的具体模块**。
2. **大纲同步修改**：在扩充单日教学内容的同时，AI **必须同步修改周度全局大纲文档（`docs/week_XX_*.md`）中对应单日的核心知识点与过关验证标准**，确保全局大纲与实际教学内容保持 100% 动态对齐。
3. **交付件全链路对齐**：扩展后的内容必须同步体现在当日的 `notes.md`（技术沉淀）、`practice.py`（TODO 模版）以及参考标准答案代码中，确保全套教学交付件的一致性与高完备度。

### 19. 包管理工具规范 (Package Management Standard)
* **包管理与虚拟环境**：本项目统一且严格使用 **`uv`**（优先使用 `uv pip install <package>` 进行极速包管理）和 **`pip`** 管理 Python 依赖与本地 `.venv` 虚拟环境。在执行任何依赖安装或环境配置命令时，必须遵守此基本原则。

### 20. 项目基础设施与环境变量配置规范 (Infrastructure & Environment Config Standard)
* **环境变量与动态配置读取**：项目的所有基础服务（LLM API、Redis、PostgreSQL 等）凭证与网络地址**严禁在代码或文档中硬编码敏感账号密码**。所有配置统一通过 `weekly/w04_prompt_and_http/utils.py` 的 `load_env_file()` 动态从本地 `.env` 文件或系统环境变量中读取：
    * **LLM API 配置变量**：`MINIMAX_API_KEY`, `MINIMAX_BASE_URL`, `MINIMAX_MODEL`
    * **Redis 配置变量**：`REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD`, `REDIS_DB`
    * **PostgreSQL 配置变量**：`POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`
* **解耦原则**：代码与规范中仅保留环境变量 Key 的契约定义，真正的敏感账号密码完全收拢在本地未纳管的 `.env` 中，确保代码安全的工业级解耦。

### 21. LLM 可靠性中间件使用规范 (LLM Reliability Middleware Standard)
* **中间件架构定位**：项目在与 `weekly/` 平级的 `middlewares/llm_reliability_adapter` 中沉淀了通用可复用的结构化解析中间件（提供双层门面架构）。
* **严禁重复堆砌解析函数**：后续所有涉及大模型（LLM）结构化提取、JSON 解析与纠错场景的代码（练习、博弈引擎、工具调用等），**严禁重新编写复杂的正则解析函数或单体 parse_xxx_json 函数**。
* **统一调用契约**：
    * **优先使用【第一门面】`parse_structured(raw_text, response_model)`**：对于已有 LLM 调用代码的场景，一律导入并使用 `from middlewares.llm_reliability_adapter import parse_structured` 一键完成 100% 可靠提纯、字符栈匹配、Pydantic 校验与 0 延迟本地修补。
    * **使用【第二门面】`UniversalAdapter`**：当需要中间件全面托管 LLM 网络请求、Level 2 带堆栈 Re-prompt 纠错重试及 Level 3 熔断降级时，使用 `UniversalAdapter(driver)`。
