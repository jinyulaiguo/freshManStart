## 📐 融合设计原则

在动手合并之前，先说清楚我做每个取舍的理由：

| 设计决策 | 选择 | 来源 | 理由 |
|----------|------|------|------|
| 是否保留 Python 基础 | ✅ 保留，占 2 周 | V1 | 异步和 Pydantic 不能跳过，否则 W5 手写 ReAct 会卡壳 |
| 是否保留 LLM 原理 | ✅ 保留，与 API 合并为 1 周 | V1 | 面试必考，但不需要独立一整周 |
| 每周驱动方式 | 项目驱动，每周有具体作品 | V2 | 项目驱动远优于知识驱动 |
| LangGraph 位置 | 放在 RAG + Memory 之后 | V2 | 先知道痛点，再学框架才能理解设计意图 |
| 是否加入 GraphRAG | ✅ 加入 | V2 | 面试差异化加分项 |
| Planning/Reflection 位置 | 独立一周，放在 LangGraph 之后 | 两版都缺 | 这是 Agent 核心能力，不能缺也不能太晚 |
| Eval 深度 | 独立一周 | V1 | V2 把 Eval 混在 W23 太弱 |
| Context Engineering | 独立一周 | V1 | 2025–2026 热门概念，值得专题 |
| 生产化框架 | Harness Engineering 三层架构 | V2 | 比 V1 零散的工程化内容更有体系 |
| 多 Agent 案例 | V2 的四种协作模式 | V2 | 覆盖更全面 |
| 面试准备启动 | W15 结束后开始 | 融合改进 | 两版都太晚 |
| 贯穿性项目 | 从阶段二开始累积 | 融合改进 | 两版都缺少主线项目 |

## 阶段一：Python + LLM 地基（第 1–5 周）

> **核心目标**：扫清语法障碍 + 理解 LLM 原理 + 不依赖框架手写 ReAct 循环

### W1：Python 核心能力（上）

**学习内容**：
- 变量、数据类型、列表/字典/集合/元组操作
- 条件判断与循环
- 函数定义、`*args` / `**kwargs`、lambda
- 闭包与装饰器原理
- 模块导入机制（import 三种方式）
- 异常处理（try/except/finally）

**重点强调**：字典操作要极度熟练；装饰器要理解到"能自己写一个带参数 of 装饰器"的程度。

**过关检验**：手写一个装饰器，能记录函数的调用参数、返回值和执行时间。

---

### W2：Python 核心能力（下）

**学习内容**：
- 面向对象：类、继承、多态、`__init__` / `__call__` / `__repr__`
- **异常处理框架进阶**：自定义异常继承体系设计、异常链传递（`raise ... from ...`）、利用 `traceback` 提取与记录调用栈
- 抽象基类（ABC）与 Protocol
- 类型注解：typing 模块、Optional / Union / Literal / Annotated
- **Pydantic**：BaseModel、Field、嵌套模型、validator、导出 JSON Schema
- **TypedDict** 与 **dataclass**（与 Pydantic 对比）
- Enum 基础用法
- **异步编程**：async/await、asyncio.run()、asyncio.gather、异步上下文管理器
- 生成器（yield）与上下文管理器（with / `__enter__` / `__exit__`）

**重点强调**：Pydantic 和异步是本周的重中之重，时间不够优先保这两个。

**过关检验**：用 Pydantic 定义一个 Tool Schema（含嵌套参数），导出 JSON Schema；用 asyncio.gather 并发调用 3 个异步函数。

---

### W3：LLM 原理 + API 调用

**学习内容**：
- Transformer 架构直觉理解（自注意力 = 每个词看其他所有词的相关性）
- 自回归生成原理（逐 token 预测下一个）
- 为什么 LLM 会幻觉（概率采样 + 无事实验证机制）
- **Tokenizer 原理与实操**（用 tiktoken 实际计算 token 数，理解中英文 token 差异）
- 上下文窗口限制的本质
- temperature / top_p 对输出的实际影响
- 熟练调用 OpenAI / DeepSeek / Qwen API
- 掌握 system / user / assistant 角色设定

**产出作品**：一个能模仿特定风格（如"鲁迅风"）的聊天机器人，并附带 token 消耗统计。

**过关检验**：能用自己的话向面试官解释"Transformer 怎么工作"和"LLM 为什么会幻觉"。

---

### W4：Prompt 工程进阶

**学习内容**：
- Chain-of-Thought（思维链）原理与实操
- Few-shot 示例设计
- 结构化输出 Prompt 设计（强制 JSON 输出）
- JSON Mode / Structured Output 的各厂商实现
- 对比不同 Prompt 结构下的准确率
- 环境管理基础（venv、pip、.env 文件、python-dotenv）
- HTTP 请求基础（requests / httpx、REST API 结构、流式响应处理）

**产出作品**：一个简历信息提取器（输入自由文本，输出标准化 JSON），并对比 3 种不同 Prompt 策略的准确率。

**过关检验**：能解释 Zero-shot vs Few-shot vs CoT 的适用场景差异。

---

### W5：手写 ReAct + Function Calling

**学习内容**：
- 精读 ReAct 原论文的核心思路
- **纯手写**（不用任何框架）实现 `while max_steps` 循环
- 手动解析 LLM 输出的 Thought / Action / Observation
- 学习各厂商的 Function Calling 规范
- 构建工具描述 JSON 数组
- 处理并行工具调用
- 实现 2–3 个自定义工具（搜索 + 天气 + 计算）

**产出作品**：一个实时资讯查询 Agent（集成搜索 + 天气 + 计算三种工具），完全不依赖 LangChain。

**✅ 里程碑一：可运行的最小 Agent**

**过关检验**：面试官问"ReAct 是怎么工作的"，能在白板上画出完整循环并解释每一步；能展示自己手写的代码。

## 阶段二：知识与记忆系统（第 6–9 周）

> **核心目标**：解决 Agent 的"知识陈旧"和"对话健忘"问题
> 
> **🔵 贯穿性项目启动**：从本阶段开始构建"**AI 研究助手**"——一个能接收研究课题、检索资料、分析文档、记住历史对话的智能助手。后续每个阶段都在给这个项目加能力。

### W6：Embedding + 向量数据库

**学习内容**：
- Embedding 模型原理（文本 → 高维向量 → 语义空间）
- 主流 Embedding 模型对比（BGE、Voyage、OpenAI Embedding）
- 向量数据库部署与操作（Milvus / Chroma / Qdrant 选一）
- 向量的 insert / search / 相似度计算
- 距离度量方式（余弦相似度 vs L2 距离）

**产出作品**：一个文本相似度搜索引擎（输入一句话，返回最相似的 5 条历史文档）。

**贯穿项目进度**：为研究助手搭建底层知识检索能力。

---

### W7：经典 RAG 架构

**学习内容**：
- **分块策略深度学习**（固定大小 vs 递归分块 vs 语义分块，至少花 1.5 天）
- 重叠（overlap）的作用与参数调优
- Retrieve → Augment → Generate 全流程
- 检索为空时的降级处理
- Source Citation（引用来源）设计

**产出作品**：一个公司内部规章制度问答 Bot（上传 PDF 即可问答，回答附带引用来源）。

**贯穿项目进度**：研究助手获得"读文档并回答问题"的能力。

---

### W8：高级检索 + GraphRAG

**学习内容**：
- 多路检索（Multi-Query Retrieval）
- Rerank 重排序（交叉编码器）
- HyDE（假设性文档嵌入）
- 知识图谱基础（实体识别、关系抽取）
- GraphRAG（微软开源方案）的社区报告生成与实践
- 对比 RAG vs GraphRAG 在复杂推理场景下的效果
- Tool Retrieval 概念（500 个工具中动态检索 3–5 个）

**产出作品**：一个小说人物关系分析器（用 GraphRAG 构建人物关系图并回答跨人物的推理问题）。

**贯穿项目进度**：研究助手支持复杂的跨文档推理。

---

### W9：多层级记忆系统

**学习内容**：
- 短期记忆（滑动窗口）
- 长期记忆（向量化摘要存储 + 召回）
- 记忆压缩与总结策略
- 跨会话持久化设计
- 记忆截断策略（Token 预算管理）

**产出作品**：一个长期陪伴式心理慰藉 Agent（能记住一个月前聊过的事情，能主动引用历史对话）。

**✅ 里程碑二：具备 RAG + Memory 的完整知识型 Agent**

**贯穿项目进度**：研究助手获得跨会话记忆能力，能记住之前的研究课题和结论。

**过关检验**：能独立搭建一个具备 RAG + Memory 的 Agent，并能解释分块策略、检索方式、记忆架构的每个设计决策。

## 阶段三：框架与核心工程（第 10–15 周）

> **核心目标**：引入工业级框架 + 补齐 Agent 核心推理能力 + 建立工程化基础
>
> **🟢 开源参与启动**：从本阶段开始阅读 LangGraph 源码，遇到 Bug 或文档问题随手提 Issue/PR。

### W10：LangGraph 核心

**前置准备**（前 2 天）：快速过一遍 LangChain Core 的基础抽象——`BaseChatModel`、`Runnable` 接口、`invoke / ainvoke / stream` 三件套。不学链式编排（已过时），只理解接口层。

**学习内容**：
- 图状态机概念
- StateGraph、Nodes、Edges 的定义
- `add_messages` 状态归约器
- TypedDict 定义 State（前呼应 W2 所学）
- 条件边（Conditional Edges）
- `recursion_limit` 防死循环

**产出作品**：用 LangGraph 重写 W5 的 ReAct Agent，并新增一个固定流程的客服机器人（意图识别 → 查询 → 回复）。

**贯穿项目进度**：研究助手迁移到 LangGraph 架构。

---

### W11：LangGraph 进阶

**学习内容**：
- 循环图设计
- HITL（人在回路）`interrupt` 断点机制
- 人工审核工具调用（"是否允许执行此操作"）
- "时间旅行"（从断点重新运行）
- 子图（Subgraph）
- Checkpoint 持久化

**产出作品**：一个需人工审批的邮件发送 Agent + 一个能自我纠正的 SQL 查询 Agent。

**贯穿项目进度**：研究助手加入人工审核节点（对重要结论让用户确认）。

---

### W12：Planning + Reflection 范式 ⭐ 两版都缺的关键补充

**学习内容**：
- **Plan-and-Execute 架构**：先生成完整计划，再分步执行
- **ReWOO**（Reasoning Without Observation）：规划与执行解耦
- **Reflexion / Self-Refine**：Agent 自我审视、从错误中学习
- LLM-as-Critic 设计模式（用一个 LLM 审查另一个 LLM 的输出）
- 对比 ReAct vs Plan-and-Execute 的适用场景
- 用 LangGraph 实现一个 Plan-and-Execute Agent

**产出作品**：一个能自动拆解复杂研究任务的 Agent——给定"分析某行业的市场趋势"，它能自动拆解为子任务，分步执行，并在执行中自我纠错。

**贯穿项目进度**：研究助手获得"规划能力"——面对复杂课题能自动拆解并分步研究。

**过关检验**：能回答"ReAct 效率低怎么办"（答：Plan-and-Execute）和"Agent 输出错了怎么办"（答：Reflexion）。

---

### W13：MCP 协议 + 标准化工具层

**学习内容**：
- Model Context Protocol 的 Client-Server 架构
- 基于官方 SDK 手写 MCP Server（暴露本地资源）
- 将 MCP 集成到 LangGraph 中
- Tool 设计最佳实践（工具描述怎么写、参数怎么设计才能让 LLM 准确调用）
- 工具动态检索（Tool Retrieval：大量工具中智能选择少量使用）

**产出作品**：一个连接本地文件系统的 MCP Server + 一套标准化的工具接入层（至少包含 5 个工具）。

**贯穿项目进度**：研究助手通过 MCP 接入搜索引擎、文件系统、数据库等外部资源。

---

### W14：Context Engineering

**学习内容**：
- 上下文分层架构（System 层 / 记忆层 / 检索层 / 对话层）
- Token 成本精确计算与预算管理
- Prompt Cache 策略（Anthropic / OpenAI 的缓存机制）
- 长对话摘要压缩
- 模型路由策略（简单任务用小模型，复杂任务用大模型）
- Fallback 策略（大模型失败时降级到小模型）

**产出作品**：优化研究助手的 Token 消耗——实现自动摘要压缩 + 模型路由模块，对比优化前后的 Token 成本（目标降低 30%+）。

**贯穿项目进度**：研究助手的运行成本大幅下降。

---

### W15：评测体系（Eval）

**学习内容**：
- Golden Dataset 构建方法（人工标注 + LLM 辅助生成）
- LLM-as-Judge 评测范式
- 评测指标设计：
  - 任务完成率
  - 工具调用准确率
  - 回答相关性（Relevance）
  - 回答忠实度（Faithfulness，是否有幻觉）
- 回归测试策略（防止"版本越改越烂"）
- 将评测接入 CI/CD（GitHub Actions 自动化）
- 评测报告自动生成

**产出作品**：一套 ≥50 条测试用例的评测集 + 自动化评测流水线（每次代码提交自动跑测试并生成报告）。

**✅ 里程碑三：具备框架 + 规划 + 评测的完整 Agent 系统**

**贯穿项目进度**：研究助手拥有完整的质量保障体系。

**过关检验**：能回答"你怎么知道你的 Agent 变好了还是变差了"。

## 阶段六：Harness Engineering（第 22–24 周）

> **核心目标**：达到"控 AI 不失控"的生产级水平
> 
> **核心公式**：`Agent = Model + Harness`
> 
> **核心哲学**：当 Agent 犯错时，不换更强的模型，而是改进 Harness。

### W22：引导层（Guides / Feedforward）— 执行前约束

**学习内容**：
- 编写 SPEC 任务定义文档
- 设置硬性 Rule 规则（如“禁止访问宿主机敏感路径”、“关键写入必须确认”）
- 封装可复用的 Skill（技能包）
- AGENTS.md 规范文档编写
- 动态激活规则（根据任务上下文按需触发，避免塞太多 Prompt）

**产出作品**：一套针对金融风控或代码执行场景的 SPEC 规范文档与约束代码。

---

### W23：约束层（Constraints）— 执行中控制

**学习内容**：
- Docker 容器作为 Agent 执行沙箱
- 细粒度文件系统权限设计
- 网络策略控制（限制 Agent 的网络访问范围）
- 关键操作强制二次授权
- Prompt Injection 防护
- 输入护栏（敏感数据过滤、意图分类拦截）
- 输出护栏（格式校验、敏感内容拦截）

**产出作品**：一个安全的隔离执行沙箱，Agent 生成的代码只能在此受限沙箱中运行并返回结果。

---

### W24：反馈层（Sensors / Feedback）— 执行后校验

**学习内容**：
- 自动验证反馈回路（单元测试 + Linter 静态扫描 + 运行日志的自动化纠错）
- 监督机制：审计 Agent 对主 Agent 的输出合规性与逻辑一致性进行校验
- Reflexion 自动化（Agent 根据错误日志自动重试 and 优化）
- 语义缓存（Semantic Cache，减少重复调用）
- Token 费用监控与预算熔断保护

**产出作品**：为“AI研究助手”完成 Harness 三层控制环的完整封包（前有约束引导，中有隔离沙箱，后有自检反馈）。

**✅ 里程碑六：Harness-Engineered 的生产级 Agent 系统**

**过关检验**：能回答"你怎么保证 Agent 不失控"，并从引导层、约束层、反馈层三个维度完整作答。

## 📊 贯穿性项目演进路线

```
W6  ┌─ 基础版 ─────── 能检索本地文档并回答问题
W7  │  + RAG ────────── 上传 PDF 即可问答，附带引用来源
W8  │  + 高级检索 ───── 支持跨文档推理
W9  │  + Memory ────── 记住之前的研究课题和结论
    │
W10 │  + LangGraph ─── 工作流编排，状态管理

    │

W17 │  + 可靠性 ────── 失败自动恢复

    │
```
