# 📅 Week 19: 多 Agent 设计模式

> **第十九周目标**：理解单 Agent 的能力边界，掌握“规划-执行-审查”三角协同架构，精通顺序链模式、层级模式与辩论/投票模式的底状态机设计，熟练编写基于消息的 Handoff (交接手) 动态状态传递算法。

## Day 127：“规划-执行-审查”三角多 Agent 架构协同逻辑
*   **核心知识点**：
    *   **角色界定**：
        - Planner-Agent：只负责任务大纲的制定；
        - Worker-Agent：根据大纲调用特定工具产出内容；
        - Reviewer-Agent：根据大纲及规范严格审计 Worker 的产出（不合格则驳回重写）。
    *   **状态共享规范**：三大 Agent 如何在同一个 State 拓扑中只修改自己有权限的 Key，防止状态写覆盖冲突。
*   **Agent 核心关联**：单个大模型又要做计划、又要做工具执行、还要自己校验自己，会由于认知过载导致极高的失误率。拆分为三个独立的专业 Agent 角色进行博弈，是攻克复杂业务（如自动化报税、高精度财务审计）的必由之路。
*   **🎯 过关验证标准**：在 LangGraph 中设计一个三节点的三角状态图，实现 Planner 规划 -> Worker 执行 -> Reviewer 审查逻辑。编写测试用例使 Reviewer 驳回 Worker 的生成物 2 次，验证状态图能自动沿路由边重试，直到合格后正常输出。

## Day 128：顺序链（Sequential）模式下 Agent 间的输入输出管道
*   **核心知识点**：
    *   **顺序链（Sequential Pattern）模型**：任务在多个 Agent 间线性流转（Agent-A $	o$ Agent-B $	o$ Agent-C），每个 Agent 都是上游输出的精炼加工者。
    *   **管道参数对齐**：上游 Agent 输出的文本，如何通过反射自动序列化为下游 Agent 的强类型 Pydantic 输入。
*   **Agent 核心关联**：用于流式作业管道（Pipeline，例如：输入中文文章 $	o$ 翻译 Agent 翻译为英文 $	o$ 润色 Agent 修正语法 $	o$ SEO Agent 生成关键字）。这种链式处理结构清晰，各节点职责极其纯净。
*   **🎯 过关验证标准**：构建一个三节点 Sequential 状态链。上游节点输出的非结构化字符串，通过 Pydantic 管道拦截，自动转换为强类型的入参结构输入到下游节点，运行测试验证全链路 0 报错。

## Day 129：层级（Hierarchical）模式下的中央协调器与分权子图
*   **核心知识点**：
    *   **中央调度器（Supervisor Agent）**：不直接执行工具，只负责解析任务并呼叫特定的子 Agent。
    *   **子图委托（Delegation）**：子 Agent 作为一个完全隔离的 Subgraph，接收指令执行完毕后，只将结果返回给 Supervisor。
    *   **状态的层级汇聚**。
*   **Agent 核心关联**：当业务庞杂到拥有 50 个工具时，任何单 Agent 都会崩溃。层级模式中，Supervisor 就像项目经理，把任务拆分后派发给专门的开发小组（子 Agent 图），实现了“复杂性的大规模分治”。
*   **🎯 过关验证标准**：手写一个层级 Agent 系统。主图包含一个 Supervisor 路由器，子图包含一个能查天气的 WebAgent 和一个能写文件的 FileAgent。测试输入“去网上查北京的天气并写入本地 weather.txt 报告”，验证 Supervisor 能精准依次呼叫两个子 Agent 协作完成。

## Day 130：辩论与投票（Debate/Voting）模式的多模型共识机制
*   **核心知识点**：
    *   **Debate 模式逻辑**：多个具有不同 System 人设的 Agents 针对同一个主题进行交替发言（正方 vs 反方 $	o$ 裁判判定）。
    *   **共识评估指标**：计算多次生成文本的余弦相关度，判定共识是否收敛。
    *   **多票汇总（Majority Voting）算法**。
*   **Agent 核心关联**：对于非结构化决策（如“这套代码是否存在架构设计硬伤？”），单一模型的局限性很大。通过多个不同模型（如 Claude 和 GPT）化身正反两方进行辩论，能够碰撞出更全面的优缺点，为人类决策提供极高质量的参考。
*   **🎯 过关验证标准**：设计一个辩论图拓扑。包含 `Agent-A (性能至上者)`、`Agent-B (安全至上者)` 和 `Judge-Agent`。针对“是否应该在系统中使用多路并发”进行两轮辩论，由 Judge 最终给出中立的判定报告并输出。

## Day 131：基于消息的 Handoff (交接手) 动态状态传递算法
*   **核心知识点**：
    *   **Handoff（交接）协议规范**：大模型决定将当前的控制权交出（如输出特殊指令 `TRANSFER_TO_AGENT(agent_name)`），携带当前未处理完的 State 参数。
    *   **动态控制转移**：条件路由边捕获 Handoff 指令，动态改变目标 Node 指向，并清除或同步临时会话历史。
*   **Agent 核心关联**：这模仿了真实客服中的“转接人工”或“转接专员”。当普通客服 Agent 发现解决不了用户退款时，触发 Handoff 动态将控制权和之前的对话快照交给退款专员 Agent 执行，实现了平滑的主体转接。
*   **🎯 过关验证标准**：在有向图中实现动态 Handoff 机制。普通问答 Agent 识别到用户提问涉及“财务安全”时，输出 `Handoff(FinanceAgent)` 指令，图自动跳转至 FinanceAgent，并带过去当前的会话状态参数。

## Day 132：多 Agent 间的共享 State 设计与数据覆盖冲突防御
*   **核心知识点**：
    *   **共享状态字典（Shared State）**：多 Agent 图中所有节点共享同一个 TypedDict。
    *   **脏写冲突（Dirty Write Conflict）**：多节点并发运行时，对同一 key 覆盖写入导致的数据丢失。
    *   **前缀保护与命名空间隔离（Namespace Isolation）**：在 State 中为每个 Agent 划分独立的数据 Key（如 `agent_a_workspace`）。
*   **Agent 核心关联**：如果多 Agent 都在写同一个全局消息列表，可能会导致对话顺序错乱和参数踩踏。通过在 State 字典中为每个 Agent 设计隔离的命名空间，能确保并发协作时的安全性。
*   **🎯 过关验证标准**：设计一个包含 Namespace 隔离的 TypedDict 状态类。并发拉起 3 个子 Agent 写入其专有工作区属性，验证最终汇聚节点的 State 中所有 Agent 的数据完整保留，无一发生数据踩踏或覆盖丢失。

## Day 133：第十九周综合实战：基于层级架构的多角色（研究员、文案写手、校对员）研报生成多 Agent系统
*   **实战任务**：**为“AI 研究助手”升级架构，打造多 Agent 级高精度研报生成系统。**
    *   **要求**：
        1. 采用层级（Hierarchical）架构设计，包含一个 Supervisor Agent 统筹；
        2. 子节点包括：① 负责爬取并召回知识的研究员（Researcher Agent，绑定 RAG 工具）；② 负责根据提纲写材料的文案写手（Writer Agent）；③ 负责审查文料合规度与错漏字并给出驳回意见的校对员（Editor Agent）；
