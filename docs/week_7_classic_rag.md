# 📅 Week 7: 经典 RAG 架构

> **第七周目标**：精通文本文档的各种切分（Chunking）算法，掌握语义相似度突变点的概念，攻克长上下文信息遗忘痛点，设计具备可信来源标引（Source Citation）的经典 RAG (检索增强生成) 管道。
> 
> **🔵 贯穿性项目进度**：本周为“**AI 研究助手**”赋予解析多格式本地 PDF 文档并进行语义切分、以及具备准确引用对照的经典 RAG 问答交互能力。

## Day 43：分块策略深度学习（一）：固定大小 vs 递归字符分块
*   **核心知识点**：
    *   **固定字数分块（Fixed-size Chunking）**的缺陷：无视自然语义边界，极易导致段落或代码函数体在中间被强行截断，破坏高维表征的相关度。
    *   **递归字符分块（Recursive Character Chunking）**的算法逻辑：定义分块大小（Chunk Size）与重叠大小（Chunk Overlap），按照 `['\n\n', '\n', ' ', '']` 分隔符列表递归尝试切分，尽量将完整的段落和句子保留在同一个块内。
    *   **Overlap（块重叠）**的平滑作用：在相邻分块间保留一定比例的冗余文本，保证跨块边缘的语义不丢失。
*   **Agent 核心关联**：分块太大会引入冗余噪音且挤爆大模型的 Context Window；分块太小会丢失必要的上下文脉络。掌握递归字符分块参数的细微调优，是保证后续向量匹配精确度的工程红线。
*   **🎯 过关验证标准**：手写一个递归分块函数，传入长篇技术文档，验证在不同 `chunk_size` 和 `chunk_overlap` 设定下，生成的 Chunk 块中是否能完美保留完整的 Python 异步函数体，而不发生代码行被暴力砍断。

## Day 44：分块策略深度学习（二）：基于语义的语义分块（Semantic Chunking）
*   **核心知识点**：
    *   **语义分块（Semantic Chunking）**的底层原理：利用相邻句子（Sentences）之间的向量语义相似度突变作为切分点，使得每个 Chunk 在语义上高度内聚。
    *   **切分阈值算法**：
        1. 将文本拆为单句并计算 Embedding；
        2. 计算相邻句子间的余弦相似度，得到差值列表；
        3. 设定统计学阈值（如相似度差值大于平均值加 1 倍标准差），在该突变点执行物理分割。
*   **Agent 核心关联**：传统的字符分块完全忽略了自然语义的段落逻辑。采用语义分块能保证灌入向量数据库的每一个 Chunk 都是一个高度纯净的“语义事实”，能大幅提升 RAG 系统的检索匹配度。
*   **🎯 过关验证标准**：实现一个 `SemanticTextSplitter`。输入一篇混合了多个话题（如“量子计算”与“中餐烹饪”）的长文章，动态计算相邻句子的 Embedding 距离，在相似度暴跌处自动断开，打印出切块后的段落分布，证明分块在话题转换点完全切分。

## Day 45：Retrieve -> Augment -> Generate 经典工作流编排
*   **核心知识点**：
    *   **RAG 经典三阶段**的 Python 原生编排：Query 向量化 $\to$ 向量库 Top-K 召回 $\to$ 拼装 Context 并渲染 Prompt $\to$ 大模型生成。
    *   **检索结果为空时的兜底逻辑（Null Result Fallback）**：当向量库返回相似度全低于阈值时，如何平滑降级，防范大模型胡说八道。
    *   **Context 注入界限**：在 System 提示词中设置严格的“类型契约”，限制模型必须只根据 Context 解答。
*   **Agent 核心关联**：RAG 是 Agent 在面临实时外部知识和专有私有库时的核心补充手段。掌握原生编排和空检索过滤，能够防止 Agent 产生顽固幻觉。
*   **🎯 过关验证标准**：编写一个通用的 `RAGPipeline`，输入用户提问，当向量库返回的最高相似度低于 0.6 时，系统自动触发“空结果降级”——直接回复“对不起，未在参考库中找到对应事实”，不进行模型调用；相似度达标时，拼装上下文完成解答。

## Day 46：基于 Context 来源的可信引用（Source Citation）与脚注机制
*   **核心知识点**：
    *   **元数据溯源（Metadata Traceability）**：在文档切分导入向量库时，每个 Chunk 必须绑定全局唯一的 `doc_id`、`source_file` 和 `page_number`。
    *   **标引指令设计（Citation Prompting）**：在 System Prompt 中强制要求大模型在生成的任何结论后，必须使用 `[doc_id:page_number]` 的格式作为脚注进行精确引用说明。
    *   **Python 引用过滤解析**：在 Python 端接收到大模型生成的文本后，利用正则表达式解析出所有的 `doc_id` 脚注，并去重提取出对应的源文件名及原始段落。
*   **Agent 核心关联**：在企业级应用中，无法给出数据源对照的 Agent 回复是完全无法上线的。可信引用机制不仅能够提高用户信任度，也是开发人员排查检索源和生成幻觉的关键监控手段。
*   **🎯 过关验证标准**：设计一个 Context 数据格式类。通过 Prompt 引导大模型在回答时，针对所引用的事实在句末加入 `[doc_id:page]` 标记。编写 Python 解析函数，在流式生成中实时捕捉并提取出大模型声明的所有引用源 ID，并在终端末尾输出可溯源的原始文档对照表。

## Day 47：长上下文（Long-Context）填充位置对召回召回率的影响（Lost in the Middle）
*   **核心知识点**：
    *   **Lost in the Middle（遗忘在中间）**现象的本质：大模型在处理包含极长上下文的输入时，其注意力分配通常集中在 Prompt 的最开始和最末尾，中间部分的召回准确率呈 U 型曲线下跌。
    *   **黄金位置重排序算法（Golden Position Reordering）**：将检索得到的 Top-K Chunk 列表，按照语义相关度从大到小交错分布在输入列表的首尾两端，中等相关度的块放置在正中间。
*   **Agent 核心关联**：在单轮决策中召回 15-20 个 Chunk 时，如果盲目按照相似度顺次拼接进 Prompt，最核心的第 3-8 个 Chunk 极易被大模型遗忘，导致 Agent 决策流中做出错误的选择。
*   **🎯 过关验证标准**：编写一个 `ReorderContextHandler`。接收一个粗筛得到的 Chunk 对象列表，运行重排逻辑：将相似度前两名的 Chunk 分别放到列表第一位和最后一位，其余以此往复，打印并验证重排后列表的位置分布完全符合“首尾最重、中间最轻”的 U 型交错规则。

## Day 48：非结构化文档多模态解析（HTML / Markdown / PDF）与元数据富集
*   **核心知识点**：
    *   **PDF 结构化解析**：使用 `pdfplumber` 或 `pypdf` 提取复杂文本和多列排版，避免段落逻辑破损。
    *   **表格提取与归一化**：将 PDF 中的表格数据提取为 Markdown 表格或标准 JSON 结构，因为大模型对结构化的 Markdown 表格具备更好的语义理解力。
    *   **元数据富集（Metadata Enrichment）**：自动计算提取时间、文档标签、分类和用户权限，并拼接到 Chunk payload 中。
*   **Agent 核心关联**：在企业生产环境中，文档多数以扫描版 PDF、带表格的 Word、或是 HTML 形式存在。设计高鲁棒性的文档多模态解析引擎，是保证输入数据不失真的工程前置防线。
*   **🎯 过关验证标准**：编写一个 `MultiFormatDocIngestor` 类。能够自动检测输入目录下的 `.pdf`、`.html`、`.txt` 文件并调用对应的解析组件。要求：对 PDF 中的表格实现高保真 Markdown 格式提取，并将文件名、页码、文件大小作为富集元数据（Metadata）统一导出。

*   **综合实战任务**：**为“AI 研究助手”实现一个具备多格式文档提取、语义切片、黄金重排、以及完美引用机制的经典问答 Bot 系统。**
