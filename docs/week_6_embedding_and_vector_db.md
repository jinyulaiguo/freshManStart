## Day 36：文本向量化（Embedding）数学直觉与表征限制
*   **核心知识点**：
    *   **余弦相似度（Cosine Similarity）**与 **欧氏距离（L2 Distance）**的底层数学公式及几何几何物理意义：余弦专注于向量方向上的相似度（不受文本长度引起的长模长影响），L2 专注于高维空间中点的绝对位移。
    *   **维度裁剪（Matryoshka Representation Learning）**：理解最新模型（如 OpenAI `text-embedding-3`）如何在不显著损害语义表达的情况下，通过损失少许边缘维度将 1536 维截断到 256 维以节省存储开销。
    *   **表征退化与歧义限制**：文本中的反讽、特定代词、长难句在映射到单一高维向量时发生的特征稀释现象。
*   **Agent 核心关联**：向量化是语义检索的基石。在构建 Agent 知识检索时，如果不理解不同 Embedding 模型的表征偏好和距离计算差异，召回的数据极易偏离正确轨道。
*   **🎯 过关验证标准**：调用本地 BGE 模型或 OpenAI 接口，对多组故意构造的带有转折、反讽以及高相似词汇的测试样本进行向量化。利用 `numpy` 离线编写函数分别计算其余弦相似度和欧氏距离，输出对比表格并定量分析表征差异。

## Day 37：Chroma / Qdrant 向量数据库原理与本地 HNSW 索引部署
*   **核心知识点**：
    *   **HNSW (Hierarchical Navigable Small World, 分层导航小世界)** 索引算法：基于跳表（Skip List）思想构建的多层图检索算法，如何通过多层逼近实现 $O(\log N)$ 复杂度的快速近邻（ANN）查找。
    *   **IVF (Inverted File Index, 倒排文件索引)** 的聚类分桶原理。
    *   **Qdrant 数据库本地 Docker 部署**：熟悉其架构、存储结构及内存限制配置。
    *   **集合（Collection）生命周期管理**：创建集合、定义向量维度、配置距离度量参数（Cosine/Euclidean）。
*   **Agent 核心关联**：当 Agent 的知识文档规模达到万级甚至百万级时，暴力相似度检索会导致决策时延暴涨。掌握 Qdrant 本地部署与 HNSW 索引参数调优（如 `m`、`ef_construct`），能够确保知识检索阶段将响应时延死死锁在个位数毫秒以内。
*   **🎯 过关验证标准**：使用 Docker 本地启动 Qdrant 实例，通过 Python SDK 编写脚本，创建符合维度的向量集合，并发写入 10,000 条测试向量，测量其 HNSW 索引构建时间，并验证服务状态。

## Day 38：相似度检索（Similarity Search）与元数据联合过滤（Metadata Filtering）
*   **核心知识点**：
    *   **元数据过滤（Metadata Filtering）**的两种实现路径：
        1. **Post-Filtering**：先向量检索出 Top-K 结果，再在 Python 侧按属性过滤（痛点：若过滤条件苛刻，最后留下的有效结果可能为零）；
        2. **Pre-Filtering**：在向量数据库底层将元数据索引与向量 HNSW 结合（Qdrant 的 Payload Indexing 机制），检索时同步过滤。
    *   **Payload 复合条件构建**：利用 Match、Range 等条件进行精确匹配和范围过滤。
*   **Agent 核心关联**：在多用户或多租户的 Agent 系统中，不同用户拥有不同的文档阅读权限。通过 Pre-Filtering 在底层把“用户权限标识”与向量搜索联合起来，能够杜绝越权检索，并避免过滤导致的检索空置。
*   **🎯 过关验证标准**：在 Qdrant 中为向量附加元数据（包含类别 `category`、读取级别 `read_level`（整数类型）、以及时间戳 `created_at`）。编写查询函数执行 Pre-Filtering，限定 `read_level <= 2` 且时间戳在特定范围，验证返回的相似向量结果 100% 符合条件，且对比 Post-Filtering 的检索吞吐量。

## Day 39：分批向量化、批量插入与大模型限流（Rate Limit）自适应保护
*   **核心知识点**：
    *   **并发批量嵌入（Batch Embedding）**：将海量文本切割为合理粒度的 batch 数组进行并行网络请求。
    *   **API 频率限制（Rate Limit, RPM/TPM）**对系统的阻断风险。
    *   **自适应指数退避（Exponential Backoff with Jitter）重试算法**：在遇到 429 报错时，延迟重试时间，并引入随机抖动以避开网络并发碰撞。
*   **Agent 核心关联**：在 Agent 大文件冷启动导入阶段，经常需要将几百万字的文本一次性向量化导入数据库。如果只是单线程阻塞调用或暴力无限并发，会瞬间被厂商 API 封锁，高可用的批量限流导入模块是生产环境必须具备的核心底座。
*   **🎯 过关验证标准**：编写一个支持并发的批量导入模块，利用 `asyncio.Semaphore` 控制 QPS，使用带有 `tenacity`（或手写退避逻辑）的修饰器拦截大模型 API 的 429 限流报错，实现指数退避重试，将 50 万字符的文本平稳、无错地导入 Qdrant 集合中。

## Day 40：非结构化文本数据清洗、规范化与字段分流
*   **核心知识点**：
    *   **非结构化清洗（Unstructured Cleaning）**：正则去除乱码、过滤特殊控制字符（如 `\x00`）、中英文多余空格及换行合并。
    *   **实体与长文本分离**：针对复杂文档中的标题区、段落正文区、页眉页脚区进行格式提取规范化。
    *   **分流逻辑设计**：将数据归一化为标准的文本块实体，提取包含块哈希（Hash）、长度、来源标识的基础字段。
*   **Agent 核心关联**：大模型和 Embedding 模型对杂乱无章的 HTML 标签、转义符和多余空行非常敏感，噪声会扭曲高维向量的几何方向。干净的清洗流水线是避免 RAG 知识检索结果发生“表征污染”的先决条件。
*   **🎯 过关验证标准**：实现一个 `CleanTextPipeline` 类，能够接收带有 HTML 标签、复杂转义符、多换行以及特殊控制字符的测试文本，将其格式化为标准的、只包含纯粹段落语义的文本块数组，并计算出每个块的唯一 SHA-256 哈希值用作去重依据。

## Day 41：跨语言检索与 Embedding 表征漂移检测
*   **核心知识点**：
    *   **多语言表征模型（Multilingual Embedding）**的向量空间映射：实现不同语种的相同含义词汇在高维空间中共享相似的几何象限。
    *   **表征漂移（Representation Drift）**：当系统的数据分布从“日常对话”切换到“专业垂直领域代码”时，向量空间相似度计算准确率下降的量化测算。
*   **Agent 核心关联**：Agent 在全球化应用场景下经常需要处理“输入中文问题，匹配英文 PDF 文档”的场景，这就要求必须掌握多语言表征模型的匹配性质，并能定量评估表征精度。
*   **🎯 过关验证标准**：使用多语言 Embedding 模型（如 `multilingual-e5` 系列），分别对 10 组中文学术句子和其对应的英文翻译句子进行向量化，离线计算两两中英对齐句子的余弦距离，分析不同语种在同一空间下的匹配相似度阈值偏好。

## Day 42：第六周综合实战：具有元数据联合索引与限流防护的高吞吐知识检索引擎
*   **综合实战任务**：**实现“AI 研究助手”的底层知识存储与联合检索服务。**
    *   **系统功能要求**：
        1. 能够监听特定文件夹下的文件变动，提取非结构化文本，并通过 Day 40 的 `CleanTextPipeline` 执行规范化清洗；
        2. 基于 Qdrant 建立本地向量数据库集合，并自动为 `doc_id`、`created_time`、`author` 字段构建 Payload 字段索引以优化检索性能；
        3. 调用 Day 39 的高并发向量化模块，利用信号量限制请求速率，具备 429 限流自适应重试（指数退避）保护，安全地将清洗后的文本向量化写入数据库；
        4. 提供标准的检索接口，支持“用户问题向量 + 元数据联合过滤（Pre-Filtering）”，能够在高并发下稳定在 15ms 内召回 Top-K 结果。
    *   **🎯 交付件**：全套知识库导入与检索脚本、Qdrant 配置文件、异常处理装饰器代码，以及包含大批量文本导入、限流触发自愈和过滤检索完整闭环的测试用例。
