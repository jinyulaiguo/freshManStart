# 📅 Week 23: Constraints 执行中控制

> **第二十三周目标**：精通 Constraints 约束层设计，掌握基于 Docker 的隔离执行沙箱（Sandbox）构建，熟练设计细粒度 Linux 权限与文件 ACL 动态控制，实现网络防火墙策略隔离以防火墙限制域名域名访问，具备运行时二次授权拦截以及 Prompt 注入攻击的动态护栏开发能力。

## Day 155：Docker 容器作为 Agent 执行沙箱（Sandbox）
*   **核心知识点**：
    *   **沙箱隔离（Sandbox Isolation）**的底层安全机制：由于大模型生成的 Python 或 Bash 代码可能包含恶意删库、系统逃逸命令，必须将其置于完全隔离的计算容器内运行。
    *   **Docker SDK for Python** 的使用：在 Python 中通过 SDK 动态拉起一个隔离的 Alpine/Ubuntu 临时容器，将生成代码送入执行，抓取标准输出后自动销毁容器。
*   **Agent 核心关联**：如果一个写代码并运行的 Agent 直接在你的宿主机（开发电脑、云端服务器）上运行 `exec()`，大模型在遭遇 Prompt 注入后，可能会被黑客指令驱使直接删除整台服务器的硬盘。沙箱隔离是防范此类灾难的唯一防线。
*   **🎯 过关验证标准**：通过 Docker SDK for Python 编写一个 `SandboxExecutor`。大模型生成的未知 Python 脚本传入该执行器，系统自动拉起一个无外网连接、无敏感环境变量的临时容器执行它，返回 stdout 文本并自动销毁容器，验证在代码中执行删库指令对宿主机毫毫无副作用。

## Day 156：细粒度 Linux 权限与文件系统读写权限（ACL）动态受限控制
*   **核心知识点**：
    *   **Linux 用户组权限控制**：利用非 root 账号（如 `nobody` 组）运行 Agent 执行进程；
    *   **ACL (Access Control Lists) 动态受限控制**：配置只允许访问特定只读数据卷 `/app/read_only`，绝对禁止写操作，仅允许在临时垃圾箱目录 `/app/tmp` 写操作。
    *   **系统调用拦截（Syscall Filtering）**：限制执行危险的 fork 导致 fork 炸弹系统卡死。
*   **Agent 核心关联**：即便在沙箱内，也可能发生文件污染。通过细粒度 ACL 权限限制，从操作系统内核级别上剥夺 Agent 修改核心配置和代码文件的能力，保证系统文件的不可篡改性。
*   **🎯 过关验证标准**：配置本地目录的读写权限。在 Python 端调用 `os.setuid` 和 `os.setgid` 降权，验证降权后的 Agent 进程执行写库或修改系统文件时被操作系统内核直接阻断（抛出 PermissionError）。

## Day 157：网络安全防护：利用 iptables / 网络防火墙限制 Agent 域名访问
*   **核心知识点**：
    *   **网络隔离防护**：利用 `iptables` 规则或 Docker Bridge 网络配置，阻断执行沙箱向公网发起不受限制的 TCP 连接。
    *   **白名单 DNS 域名劫持过滤**：只允许 Agent 访问特定的第三方 API 域名（如 `https://api.openai.com`），拦截一切向未知黑客服务器（C2 服务器）发送的数据包以防敏感数据泄露。
*   **Agent 核心关联**：Prompt 注入攻击的一个核心手段，是引诱 Agent 去读取本地密钥，随后发送一条 HTTP 请求将密钥泄露至黑客的公网服务器。限制网络白名单能彻底切断数据外泄的数据通道。
*   **🎯 过关验证标准**：配置沙箱网络。在沙箱容器内运行一段尝试向 `http://baidu.com` 和 `https://api.openai.com` 发起请求的代码。验证百度请求由于防火墙规则被物理超时丢弃，而 OpenAI 请求能够正常建立连接并返回。

## Day 158：运行时二次授权（Two-Factor Authentication）的中断机制
*   **核心知识点**：
    *   **运行时二次授权（2FA）**：针对特定的敏感或高风险的操作节点，在执行前通过邮件、短信、或是手机 Authenticator 强行触发二次授权。
    *   **双路消息通信机制**：图状态机发送授权事件并阻断进入 `WAIT_FOR_AUTH` 状态，等待外部的授权批准通知后才继续流转。
*   **Agent 核心关联**：对于修改核心系统配置、划拨资金、或是发布代码到生产分支等操作，必须强制加入 2FA。这是物理层面上防止 Agent “自作主张”发生误操作的保底机制。
*   **🎯 过关验证标准**：设计一个带 2FA 的 LangGraph。当执行 `deploy_code` 时，向终端（模拟发送 2FA 消息）发送一条一次性六位 Token 验证请求并中断挂起。系统只在接收到正确的动态验证码后才放行该节点，否则超时回滚并告警。

## Day 159：Prompt 注入（Prompt Injection）攻击的运行时动态检测护栏
*   **核心知识点**：
    *   **运行时动态注入检测**：利用专门轻量级小模型或特定的分类器对当前输入（RAG 召回的文本、用户最新一轮提问）进行语义特征检测。
    *   **检测指标**：判定是否包含“Ignore prior instructions / 忽略之前的所有系统命令”等特征字符及变义逃逸语法。
*   **Agent 核心关联**：这是防范 Agent “被洗脑”的核心护栏。大模型容易被长上下文中的注入攻击所蛊惑，通过在运行时持续进行注入扫描，能够在发生欺骗性提示词输入时第一时间截断请求。
*   **🎯 过关验证标准**：实现一个 `PromptInjectionDetector`。向其输入各种变异的绕过 Prompt。分类器必须以大于 95% 的准确度识别并抛出 `InjectionAttackException` 强行阻断请求。

## Day 160：输入输出双向敏感数据脱敏（PII 数据脱敏）护栏
*   **核心知识点**：
    *   **PII (Personally Identifiable Information) 敏感数据脱敏**：包含身份证号、银行卡号、手机号、企业核心私钥等。
    *   **输入过滤脱敏**：发送请求前，利用正则或命名实体识别（NER）对敏感数据执行 Mask 掩码处理（如将身份证号替换为 `[ID_MASK]`）。
    *   **输出反向脱敏**：大模型生成回复后，在向用户吐出前，检查并阻断任何未经授权的敏感机密外泄。
*   **Agent 核心关联**：保护企业数据资产与用户隐私安全。防范由于大模型幻觉或指令逃逸，意外将其他租户的身份证号或核心 API key 顺带着吐给当前用户。
*   **🎯 过关验证标准**：编写一个 `PIIDataGuard` 双向过滤器。当用户提问包含手机号时，输入前自动脱敏为 `[PHONE_MASK]` 并发给模型；当模型生成的文本中意外包含数据库密码字段时，输出前自动强行替换为 `[CONFIDENTIAL_MASK]` 或直接抛错拦截。

## Day 161：第二十三周综合实战：在容器化隔离沙箱内安全执行未知代码并自动防御逃逸的 Constraints 系统
*   **实战任务**：**为“AI 研究助手”实现极其安全的 Constraints（执行中约束）物理控制环。**
    *   **要求**：
        1. 凡是执行大模型生成的 Python 或 Bash 脚本，必须调用 Day 155 的 `SandboxExecutor`，置于 Docker 临时隔离容器内运行，限死无外网连接；
        2. 对宿主机挂载的数据卷设置细粒度的只读/垃圾箱 ACL 权限，限制 uid 降权运行；
        3. 启用 Day 159 的 `PromptInjectionDetector` 进行运行时长文本输入扫描，一旦遇到注入欺骗立即拦截；
        4. 启用 Day 160 的 `PIIDataGuard` 模块，双向拦截输入输出中的敏感凭证与手机号码；
        5. 在进行数据库清空等敏感删除操作前，挂起图拓扑并发出运行时二次授权（2FA）校验，只有人工输入正确动态码后才真正生效。
