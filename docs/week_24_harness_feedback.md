# 📅 Week 24: Sensors / Feedback 执行后校验

> **第二十四周目标**：精通 Sensors 审计层与反馈层设计，掌握基于 Linter 与单元测试的执行后自动验证反馈回路，设计双大模型审计合规性校验机制，实现基于运行时错误的 Self-Correction 局部自动重试反思，具备实现语义缓存（Semantic Cache）及 Budget Breaker 熔断保护的能力。

## Day 162：执行后自动验证反馈回路：单元测试与 Linter 静态扫描
*   **核心知识点**：
    *   **代码静态分析 Linter**：在 Agent 生成代码后，利用 Python 内置的 `ast` 模块进行语法树解析，或者动态调度 `ruff` / `pylint` 对代码质量进行静态扫描。
    *   **动态单元测试验证（Unit Testing Loop）**：在沙箱内运行生成的配套单元测试，收集测试结果。
    *   **反馈回路闭环**：解析 Linter 和 unittest 返回的 stderr/JSON 报告，形成格式化错误文本喂回。
*   **Agent 核心关联**：如果一个写代码 Agent 只是负责生成而不进行验证，其代码能跑通的概率低于 60%。通过执行后拉起 Linter 与单元测试自检，不合格则打回重写，能保证输出代码 100% 具备运行能力。
*   **🎯 过关验证标准**：编写一个 `LinterFeedbackLoop`。自动解析大模型输出的文件，调用 ruff 进行静态扫描，若存在 UndefinedName 等硬伤，自动提取报错行号与原因喂回给大模型进行纠错。

## Day 163：监督机制：审计 Agent 对主 Agent 执行结果的合规性检验
*   **核心知识点**：
    *   **审计监督（Audit Agent / Sensor）**：在主 Agent 执行完任务（如写出了一份销售合同）后，不直接返回，而是将生成物分发给审计 Agent 进行合规性检查。
    *   **合规性细则校验**：审计模型检查是否包含隐私条款、是否包含特定的免责声明等硬性 SPEC 规范。
*   **Agent 核心关联**：主 Agent 生成时往往容易丢三落四。通过引入一个只看合规规范的审计 Sensor，像专业的合规部门一样把关，能够彻底消灭 Agent 回复里的法务与合规风险。
*   **🎯 过关验证标准**：构建主 Agent 节点与审计 Sensor 节点。主 Agent 生成一封促销邮件，审计 Sensor 自动检查是否包含“退订指引”和“防诈骗警示”，一旦缺漏，将具体缺失项作为报错反馈回主 Agent 重写。

## Day 164：基于运行时错误的 Self-Correction 局部自动重试反思
*   **核心知识点**：
    *   **Self-Correction 自愈控制环**：主控制流中对 Sensor 抛出的异常进行归类。
    *   **反思 Prompt 构造**：将发生报错的代码行、完整报错日志（如 FileNotFoundError）、以及反思提示（Thought: 为什么找不到文件，是不是目录拼错了？）打包组装。
    *   **局部重试步数控制**（设置最大纠错次数为 3 次，防止死锁）。
*   **Agent 核心关联**：自愈能力是生产级 Agent 不频繁抛 500 的生命支柱。如果大模型误判了当前的文件路径导致报错，系统捕获该报错后并不退出，而是给大模型一次重新调整参数或代码的机会。
*   **🎯 过关验证标准**：实现一个包含自愈重试的控制 Node。注册一个故意在特定路径找不到文件抛错的工具，测试大模型首次调用报错后，控制流能成功引导模型分析报错日志，自愈纠正路径后重新调用成功。

## Day 165：语义缓存（Semantic Cache）对输出结果的降本与加速
*   **核心知识点**：
    *   **语义缓存（Semantic Cache）原理**：对用户的 Query 进行向量化，在 Redis 语义空间中计算余弦相似度。若匹配到极其相似的历史 Query（相似度 >= 0.96），直接返回其在 SQLite/Redis 中存储的历史 Observation 答案，不调用大模型。
    *   **缓存淘汰策略（TTL 与最大内存限制）**。
*   **Agent 核心关联**：对于高频重复的查询（如“如何报销差旅费”），如果每次都走一遍 RAG 和大模型生成，会耗费昂贵的 Token 费用和数秒的时延。语义缓存能实现 5ms 内零成本的闪电般响应。
*   **🎯 过关验证标准**：基于 Redis 和向量距离手写一个 `SemanticCacheManager`。用户第一次提问“怎样走请假审批流程？”触发完整生成并存入缓存；用户第二次输入高度形似的“请假审批流程要怎么走？”时，直接命中语义缓存，不请求 LLM 并返回相同答案，时延小于 8ms。

## Day 166：运行期 Token 费用监控与实时熔断保护（Budget Breaker）
*   **核心知识点**：
    *   **运行时 Token 监控（Runtime Token Sensor）**：在每次 Node 执行大模型调用时，实时计算累计的 Token 总量与预计产生的美元金额。
    *   **熔断拦截（Budget Breaker）**：如果当前会话执行所产生的累积计费超出了预设的安全额度（如单个任务超出 0.5 美元），实时触发 BudgetExceededException 强行终止，防止 Agent 暴走产生天价账单。
*   **Agent 核心关联**：防范由于代码 bug 或大模型幻觉，导致多 Agent 之间发生循环博弈和无限对话，防止产生令企业无法承受的恶意计费账单。
*   **🎯 过关验证标准**：在图状态机中加入 `BudgetBreaker` 传感器。设定硬限额度为 0.05 美元。运行一个包含死循环工具调用的图，验证在到达此累积费用阈值的刹那，图状态机抛出异常强行中止，并输出本次运行的费用审计明细。

## Day 167：持续反馈数据清洗并转换并用于微调（Fine-Tuning）准备
*   **核心知识点**：
    *   **用户反馈收集（User Feedback Collection）**：收集用户的赞（Thumbs Up） / 踩（Thumbs Down）以及人工修改过的最终正确回复。
    *   **数据清洗（Log Parsing）**：将运行轨迹 trace 中的 JSON-RPC payload 清洗干净，排除掉隐私脱敏数据，转换为标准的 Chat 结构 `{"messages": [...]}`。
    *   **FT 训练数据导出（Export for Fine-Tuning）**：导出符合各厂商（如 OpenAI JSONL）微调规范的数据。
*   **Agent 核心关联**：这是 Agent 持续进化的飞轮。收集用户反馈的成功 Trace 数据，定期微调小模型，能让小模型在特定任务上获得与大模型比肩的高准确率，从而实现生产环境大降本。
*   **🎯 过关验证标准**：编写一个 `微调数据生成器` 脚本。能够提取 SQLite 中的成功多轮会话日志，调用 Day 160 的数据脱敏模块过滤敏感 PII 信息，自动转换为 OpenAI fine-tuning 规范的 JSONL 训练集文件并保存。
