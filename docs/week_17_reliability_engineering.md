## Day 113：重试策略与带抖动的指数退避（Jittered Backoff）实现
*   **核心知识点**：
    *   **指数退避（Exponential Backoff）公式**：退避延迟时间随着失败次数递增，$t = \min(t_{\max}, t_{base} 	imes 2^{	ext{retries}})$。
    *   **抖动（Jitter）的数学意义**：在退避时间中加入随机偏移量（如随机数乘延迟时间），防止大量的失败请求在相同的退避时间点并发请求，形成二次雪崩（Thundering Herd Problem）。
    *   **重试例外过滤**：只对 429、503 等暂时性异常重试，对 401（未授权）、400（参数错误）等永久性异常直接报错拦截。
*   **Agent 核心关联**：网络抖动和 API 短期不可用是家常便饭。具备带抖动的指数退避重试，能让 Agent 在面对网络微小波动时具有自我修复能力，不发生频繁报错。
*   **🎯 过关验证标准**：手写一个异步重试装饰器 `@async_retry`，内嵌带抖动的指数退避算法。对一个会随机抛出 ConnectionError 的模拟请求函数进行测试，验证在经历指数增长的间隔重试后，程序成功自愈并返回正确结果。

## Day 114：基于 Python 异步连接池的高频限流（Rate Limiting）令牌桶实现
*   **核心知识点**：
    *   **令牌桶（Token Bucket）算法**：令牌以恒定速度注入桶中，请求必须拿到令牌才被放行。
    *   **漏桶（Leaky Bucket）算法**。
    *   **Python 协程级限流**：使用 `asyncio` 锁与时钟机制设计非阻塞的速率限制器（Rate Limiter）。
*   **Agent 核心关联**：大模型 API 服务商对每个账户的 RPM (每分钟请求数) 都有死限。Agent 并发处理多用户任务时，如果不主动在本地客户端进行令牌桶限流，会导致后续大量请求触发 429 报错，导致整个工作流瘫痪。
*   **🎯 过关验证标准**：实现一个异步 `TokenBucketRateLimiter` 协程安全类，设定速率为 5 次请求/秒。并发启动 20 个协程请求该限流器，验证这 20 个协程最终以严格的每秒不超过 5 个的平滑速率被有序放行。

## Day 115：状态机熔断器（Circuit Breaker）的异步反射式实现
*   **核心知识点**：
    *   **熔断器三大状态管理**：
        - Closed：系统正常，记录错误率；
        - Open：当连续失败数达到阈值，直接阻断一切真实调用，快速抛出 CircuitOpenException 返回兜底降级；
        - Half-Open：熔断期过后放行少量请求进行健康状态探测。
    *   **异步反射自愈**：如何用 Python 装饰器模式优雅拦截异常并自动驱动状态转移。
*   **Agent 核心关联**：如果某个外部搜索工具或数据库 API 发生了雪崩，Agent 盲目反复发起请求，不仅会浪费大模型在等待超时期间的宝贵时间，还会加剧外部系统的瘫痪。熔断器能快速拦截并提供优雅降级。
*   **🎯 过关验证标准**：手写一个类 `CircuitBreaker`。修饰一个不可用的服务，验证当连续报错 3 次后熔断器切为 Open；等待 5 秒冷却期后切为 Half-Open，此时传入一个成功调用，验证熔断器能自动自愈恢复到 Closed 状态。

## Day 116：工具调用部分失败的 Fallback 降级与部分数据妥协
*   **核心知识点**：
    *   **局部故障降级（Graceful Degradation）**：多路工具并行执行时，某一个边缘工具运行崩溃，系统进行非破坏性降级。
    *   **占位符与降级默认数据**：当数据库不可读时，返回缓存的冷数据或默认空对象，并在 Observation 中标记 `Status: DEGRADED`，引导模型了解当前获取的数据是有限的。
*   **Agent 核心关联**：高鲁棒性的 Agent 不能因为一次无关紧要的天气工具超时就直接在前端给用户吐出 500 错误。允许部分失败和部分数据妥协，是确保生产服务可用率的常规工程策略。
*   **🎯 过关验证标准**：编写一个 RAG 检索 Node。当向量检索数据库不可用时（模拟抛出异常），节点自动进行 Fallback，返回本地备用 Mock 文本并加入降级声明，确保 LangGraph 能够继续流转直到最终输出。

## Day 117：工具幂等性（Idempotency）与 Session 事务 ID
*   **核心知识点**：
    *   **幂等性（Idempotency）**的定义：多次执行相同的操作，产生与单次执行完全相同的副作用结果。
    *   **幂等键（Idempotency Key）设计**：在写入操作中携带唯一的 Transaction ID（通常是 session_id + step_number 组合的哈希值）。
    *   **基于 Redis / 内存缓存的排重锁**。
*   **Agent 核心关联**：大模型由于超时重试或网络丢包，可能会对同一个转账或写库指令重复发送 Action。如果工具不具备幂等性检验，会导致严重的业务灾难（如二次付款、重复生成相同订单）。
*   **🎯 过关验证标准**：编写一个支持幂等性的“订单创建工具”函数，结合 Redis 写入事务锁。使用相同的幂等键连续发起 3 次并发调用，验证除了首次调用成功创建订单外，后 2 次调用均直接返回已创建的订单详情，不产生重复写库行为。

## Day 118：超时处理与 asyncio.wait_for 超时拦截重试
*   **核心知识点**：
    *   **超时硬中断**：使用 `asyncio.wait_for` 强行包裹异步网络 IO。
    *   **TimeoutError 的安全捕获**：捕获超时异常后，进行网络套接字的安全注销与连接池释放。
    *   **超时后自动回退重试**。
*   **Agent 核心关联**：如果不对工具或模型请求设置超时限制，一旦外部接口卡死，Agent 进程就会陷入永久阻塞挂起状态，占用宝贵的系统连接句柄，引发系统缓慢崩盘。
*   **🎯 过关验证标准**：编写一个网络请求包装器，设置最大超时时间为 2 秒。请求一个故意延迟 10 秒返回的模拟接口，验证包装器在第 2 秒瞬间抛出 TimeoutError，释放连接，并触发 Fallback 备用策略。
