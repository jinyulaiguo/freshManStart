## Day 50：多路检索（Multi-Query）与 Query 变形（Query Rewrite）
*   **核心知识点**：
    *   **多路检索（Multi-Query）**的本质：解决用户原始提问表达过于单一、词汇量受限而导致的语义相似度检索漏召回痛点。
    *   **Query Rewrite（查询改写）**：利用大模型将一条简短的用户指令，在保留原意的前提下，衍生重写为 3-5 句表达各异的变体问题（从不同的词汇和句式侧重去扩充检索词表）。
    *   **并行检索与结果合并去重**：异步并发请求向量库召回各路结果，利用文档哈希（SHA-256）执行合并去重（Union）。
*   **Agent 核心关联**：用户在使用 Agent 时，其输入的 Prompts 往往十分简短且缺乏专业词汇。若直接向量检索，极易错失隐藏在不同表达下的知识点。Query Rewrite 是提升检索系统召回上限的关键防守手段。
*   **🎯 过关验证标准**：编写一个 `QueryRewriter`。输入一个问题，调用大模型异步生成 3 句改写问题，并行去 Qdrant 执行向量检索，将所有返回的 Chunks 根据 `hash` 字段去重合并，并输出去重前后的 Chunk 数量与召回对比。

## Day 51：重排序模型（Rerank）与交叉编码器（Cross-Encoder）
*   **核心知识点**：
    *   **双编码器（Bi-Encoder, 如向量检索）**的局限性：将 Query 和 Document 独立向量化，只在最后通过简单的内积计算距离，丢失了微观语义交互（算力低，速度快，适合初筛）。
    *   **交叉编码器（Cross-Encoder, 如 Reranker）**的原理：将 Query 和 Document 同时输入同一个 Transformer 中进行全交互自注意力注意力计算（计算量大，打分极其精准，适合精筛）。
    *   **本地 Rerank 部署与打分**：使用 HuggingFace 部署本地重排序模型（如 `bge-reranker`），或对接 Cohere 等 Rerank API。
*   **Agent 核心关联**：向量相似度检索容易召回大量仅词汇相似但实际上逻辑无关的 Chunk，干扰 Agent 的后续判断。通过 Rerank 在初筛召回的 Top-50 数据中精准筛选出打分最高的 Top-5，能够显著提升 RAG 的最终推理精度。
*   **🎯 过关验证标准**：在本地加载 `BAAI/bge-reranker-base` 权重。编写一个 `RerankHandler`，接收向量数据库粗筛召回的 30 个 Chunk，让重排序模型重新打分，高亮输出打分前后的 Chunk 排序和打分分数变化，并丢弃打分低于 0.3 的无效 Chunk。

## Day 52：假设性文档嵌入（HyDE, Hypothetical Document Embeddings）
*   **核心知识点**：
    *   **HyDE 算法逻辑**：针对用户指令，首先使用大模型生成一篇“假设性、理论上正确”的模拟回答（Hypothetical Document），随后使用该模拟回答的 Embedding 向量去向量库检索真实的匹配 Chunk。
    *   **语义对齐瓶颈**：提问（Question）与回答（Answer）在向量空间中的拓扑分布常常不吻合，通过 HyDE 转换为 Answer 对 Answer，能大幅提升向量相似度匹配度。
*   **Agent 核心关联**：当用户提问属于“如何解决报错 X”等问句，而参考库中文档都是“X 报错的解决方法是 Y”的陈述句时，直接检索命中率极低。利用 HyDE 能够跨越“问-答”向量象限鸿沟。
*   **🎯 过关验证标准**：实现 `HyDEPipeline` 流程。首先输入复杂的问题，调用大模型生成一篇假设性的解答短文，再将其向量化去 Qdrant 中执行检索，对比不使用 HyDE 时普通检索召回的 Top-5 语义契合度差异。

## Day 53：知识图谱（Knowledge Graph）建模与实体关系提取
*   **核心知识点**：
    *   **图数据库与属性图建模（Property Graph）**：顶点（Entity, 带有标签和属性）与边（Relationship, 带有方向和属性）的定义。
    *   **三元组（Triples）的动态提取**：通过 Prompt 引导大模型提取非结构化文本中的主-谓-宾关系（Subject-Predicate-Object），并解决实体消歧（Entity Resolution，即将“张总”和“张三”识别为同一顶点）问题。
    *   **邻接表（Adjacency List）数据结构**：在 Python 侧构建内存图模型。
*   **Agent 核心关联**：传统的向量分块检索只能把相关的孤立文本抓出来，无法理解文本中复杂的实体交织关系。知识图谱建模为 Agent 提供了严密的因果与关系网络骨干。
*   **🎯 过关验证标准**：设计一套高保真三元组提取 Prompt 模版。输入一段充满复杂人际关系和地点的悬疑小说章节，让大模型准确提取出实体和关系，在 Python 侧解析为标准的图结构（Nodes & Edges）并打印出来。

## Day 54：微软 GraphRAG 框架机制分析与社区版实践
*   **核心知识点**：
    *   **GraphRAG 核心论文机制**：分析其如何通过 Chunks 提取实体 $\to$ 构建图谱 $\to$ 使用莱顿算法（Leiden）进行社区检测聚类（Graph Clustering） $\to$ 并发生成各个社区层级的“社区报告（Community Summaries）”的过程。
    *   **Local Search 检索逻辑**：结合向量检索实体与图谱邻接节点进行微观局部查询。
    *   **Global Search 检索逻辑**：直接聚合高层级的社区报告进行宏观全局推理。
*   **Agent 核心关联**：普通 RAG 无法回答宏观总结性问题（如“这 20 万字文档里反映了系统什么核心安全缺陷？”），因为相关信息散落在所有块中。GraphRAG 提供的社区总结机制是 Agent 具备“全局概览能力”的唯一突破口。
*   **🎯 过关验证标准**：本地搭建并运行 GraphRAG 运行流（或自写一个精简的基于图聚类报告汇总脚本），输入测试文档集，调用其 Local Search 与 Global Search 接口，比较两者对跨章节全局性推理问题回答的质量和完整度。

## Day 55：工具智能检索（Tool Retrieval / Dynamic Tool Dispatching）
*   **核心知识点**：
    *   **工具向量化（Tool Embedding）**：将每个本地/远程 API 的 Pydantic JSON Schema 转化为文本描述，并向量化存入专属集合。
    *   **动态工具分发（Dynamic Dispatching）**：在 Agent 执行 ReAct 循环前，先根据用户指令检索向量库，只召回最匹配的 3-5 个工具加载进 LLM 的 `tools` 列表。
    *   **多路召回与精确匹配**：结合 Tool 名称硬匹配和语义软匹配。
*   **Agent 核心关联**：当 Agent 具备上百个 API 工具时，如果一股脑全部塞给模型，会耗尽上下文空间，并导致模型决策严重错乱（Hallucinate Tool Calls）。工具动态检索是超大规模 Agent 系统上线的基石。
*   **🎯 过关验证标准**：构建一个包含 30 个不同维度工具描述的 `ToolRegistry`。编写一个工具检索器，当用户输入“帮我把这个文件上传并翻译成法语”时，系统从 Qdrant 中向量召回最匹配的 3 个工具（如：文件读取工具、HTTP上传工具、翻译工具）加载进执行链，并打印出召回的准确率和时延。

## Day 56：第八周综合实战：跨小说章节的高级逻辑推理分析器（多路 Rerank + 知识三元组）
*   **综合实战任务**：**为“AI 研究助手”实现高级检索与实体图谱推理引擎。**
    *   **系统功能要求**：
        1. 接收用户的推理提问（例如：“在这部小说中，与主角李四关系最亲密且共同在 A 地点出现过的反派是谁？”）；
        2. 首先在底层触发 Day 50 的 Multi-Query 和 Day 52 的 HyDE 进行假设性相似度粗筛；
        3. 利用 Day 51 的本地 `RerankHandler` 对初筛召回的 50 个 Chunk 进行高精度 Cross-Encoder 重打分，保留 Top-10；
        4. 对 Top-10 文本进行三元组动态抽取并构建内存图拓扑结构，自动沿“实体-地点-关系-关联人”的路径进行跨节点关联；
        5. 将图谱关联到的上下文事实与重排后的文本块一并拼装，调用大模型输出包含推理路径的最终分析报告。
    *   **🎯 交付件**：Multi-Query 检索链代码、HyDE 管道实现、本地 Rerank 接入脚本、图谱拓扑建模类、单元测试，以及演示运行的高精度推理日志。
