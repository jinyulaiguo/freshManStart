# 📅 Week 22: Guides 执行前约束

> **第二十二周目标**：精通 Guides 引导层的架构设计，掌握 SPEC 引导规范文档的编写与运行时解析，能够编写硬性规则（Hard Rules）静态过滤器，设计动态规则激活引擎，具备在执行前对 Agent 注入强约束指令的能力。
> 
> **核心哲学**：当 Agent 犯错时，不盲目更换更强的模型，而是改进控制护栏（Harness）。
> 
> **核心公式**：$Agent = Model + Harness$。

## Day 148：SPEC 引导规范文档的结构设计与解析器
*   **核心知识点**：
    *   **SPEC 规范定义**：在 Agent 执行前，定义一套规范其输入、可调用范围、输出格式的 YAML / Markdown 说明书文件。
    *   **运行时 SPEC 解析**：用 Python 编写解析器，在 Agent 启动时解析该文件，将其转化为静态属性配置并注入 System Prompt。
*   **Agent 核心关联**：普通的 Prompt 注入方式缺乏统一文档约束，多成员协作时难以管理。通过规范的 SPEC 文档，能以工程化的方式动态声明 Agent 的“工作执照和职权范围”。
*   **🎯 过关验证标准**：设计一个 YAML 格式的 `AgentSPEC` 引导文档（声明 Agent 仅限读写 `/app/data` 目录）。编写 Python 解析器将此 SPEC 读入内存，成功将其转化为 System 区域的物理 Prompt 注入。

## Day 149：执行前硬性规则（Hard Rules）的预匹配与静态检查
*   **核心知识点**：
    *   **Hard Rules 静态过滤器**：在 Agent 真正接收指令前，对指令进行前置的安全与规则扫描（例如检查指令是否包含 `rm -rf` 等破坏性字符、是否包含系统保留字）。
    *   **基于 AST（抽象语法树）的静态代码扫描初步**：如果用户要求 Agent 写代码，前置扫描其要导入的敏感包。
*   **Agent 核心关联**：大模型逻辑存在“指令逃逸”的可能性，无法在 Prompt 层面保证 100% 安全。通过在 Python 侧设计硬性的物理过滤器，能在执行前直接拦截掉一切带有物理危险的恶意用户指令，保护宿主系统安全。
*   **🎯 过关验证标准**：手写一个 `HardRuleFilter`。输入一系列测试指令，验证当指令包含“修改宿主机密码”、“读取密钥文件”等静态关键词或特定正则时，过滤器在 0.1ms 内直接抛出 `SecurityRuleViolationException`，强行拦截请求，不发送给 LLM。

## Day 150：可复用技能（Skill Packages）的封装与动态调用注册
*   **核心知识点**：
    *   **Skill 技能包**：将常用的工具链组合、特定的 Prompt 子段落以及固定的控制流，封装为一个开箱即用的“.skill”模块。
    *   **运行时动态加载**：Agent 根据 SPEC 声明，在执行前动态 load 对应的技能包绑定到状态机节点上。
*   **Agent 核心关联**：避免了为每个 Agent 重复手写工具交互代码。把常用的“生成 PDF”、“发送 Slack”等逻辑封装为标准 Skill，能实现积木式拼装 Agent 的高级工程架构。
*   **🎯 过关验证标准**：将“读取网页内容并生成 Markdown 表格”这一复合流程封装为一个名为 `WebTableExtractorSkill` 的技能类。通过 SPEC 配置文件声明挂载它，验证 Agent 能成功动态读取此技能并在运行时正确调度。

## Day 151：Workspace 级别 AGENTS.md 行为规范文档渲染
*   **核心知识点**：
    *   **AGENTS.md 规范格式**：声明项目内所有的全局设计准则、敏感操作二次授权清单以及错误码标准。
    *   **Jinja2 动态渲染**：将 `AGENTS.md` 的内容在部署阶段动态编译并注入到大模型的底层 System 空间内，保证文档与代码规范的单一事实来源（Single Source of Truth）。
*   **Agent 核心关联**：如果规范变了，我们不应该去逐个修改几十个 Agent 的 System Prompt，而应该只在全局的 `AGENTS.md` 里修改，由编译流水线自动渲染并分发到所有 Agent 实例，是标准的 CI/CD 工程化管理手段。
*   **🎯 过关验证标准**：设计一个 `AGENTS.md` 规范。编写编译渲染脚本，能够在 Agent 初始化时自动读取此文件，将最新的“二次授权操作列表”提取并动态注入到所有正在实例化的 Agent System Prompt 中。

## Day 152：动态规则过滤：根据任务上下文按需激活规则
*   **核心知识点**：
    *   **规则冗余痛点**：把 100 条安全规则全部塞进 System Prompt，会导致大模型的注意力严重稀释，且极高浪费上下文费用。
    *   **动态激活规则引擎**：先由分类模型提取用户指令类别，只检索并注入当前任务类别所需的 3-5 条硬性规则（如仅在处理“支付”时激活“大额转账确认”规则）。
*   **Agent 核心关联**：通过“规则按需激活”，既确保了当前场景的强安全约束，又将 System 提示词长度压到了极限，大幅提升时延表现并降本。
*   **🎯 过关验证标准**：设计 5 个不同类别的规则库。编写 `RuleActivator`，当用户输入“查询余额”时，只激活并注入“余额安全审计”规则，验证其他无关联的“退单流程”等规则 100% 不会被拼入 Prompt 发送。

## Day 153：工具入参的前置安全检查与防御性类型强转
*   **核心知识点**：
    *   **入参前置检查拦截器**：大模型生成工具调用参数后，在执行前，利用 Pydantic 或自定义类型校验器进行严格的值范围校验（如 `amount <= 5000`）。
    *   **防御性强转（Defensive Cast）**：若模型将参数类型传为浮点字符串而工具要求整型，拦截器执行安全的自动转换。
*   **Agent 核心关联**：虽然大模型经过了类型校验，但其输出仍偶发超出正常阈值的奇异值（如负数转账金额）。前置入参检查是在工具物理执行前的最后一道软件边界盾牌。
*   **🎯 过关验证标准**：为一个转账工具编写前置安全检查器。当大模型生成的转账金额参数为负数、或者超出限制阈值 10,000 时，检查器在调用本地函数前直接抛出 `ParameterOutOfRangeException` 并将报错返回给模型。
