2. 检索工程六大核心优化阶段

Phase 01 / Day 50
优化 Query：查询改写与多路并行
崩溃隐患 & 痛点场景
用户不会描述问题。 用户输入的原始提问往往简短、主观。例如提问“为什么 Python 很慢？”，而专业知识库里的记录是“CPython 的全局解释器锁（GIL）”。这两者几乎没有共同词汇，向量相似度极低，导致检索严重漏召回（Recall 下降）。
技术解决方案
利用 LLM 将单句原始 Query 在保留原意的前提下，重写衍生出 3-5 句句式和词汇偏好各异的变体问题（Query Rewrite），并发请求向量库执行多路检索（Multi-Query），最终采用文档哈希（SHA-256）进行合并去重。这扩大了相似度匹配的词表覆盖。

4. 真正建立的工程思维：从 Prompt 到 Context Engineering

Prompt Engineering (提示词工程)
  • 研究焦点：如何让模型更好地思考 (Thinking)。
  • 局限性：即使模型具备逻辑推导能力，如果灌入的上下文是残缺的、有噪的或无关的，推理结果依然会产生幻觉。
  • 定位：属于模型计算的“内燃机优化”。
Retrieval & Context Engineering (上下文工程)
  • 研究焦点：如何让模型获得完美的上下文 (Context)。
  • 核心逻辑：现代 Agent 的实际表现高度依赖于 Context 的纯度和结构。在推理发生前，就动态为 LLM 裁剪、组织好符合高阶任务所需的图谱知识与工具列表。
  • 定位：这是从“优化模型”走向“优化系统”的质变，也是现代工业级 Agent 落地的底座。

5. Week 8 核心技术名词释义

核心技术名词 技术标签 工业级作用与工程实质
Query Rewrite Query 优化 保持原意的情况下对 Query 进行句式/词表衍生，解决用户词汇表达单一导致的漏检索。
Multi-Query 检索并发 并行生成多个改写 Query 执行并发向量库查找，合并各路结果，提升 Recall。
Bi-Encoder 粗排双编码 Query 与 Doc 独立向量化，计算效率极高，空间相似度只进行余弦相似度等浅层运算，适合初筛。
Cross-Encoder 精排交叉编码 Query 与 Doc 组合成单文本输入 Transformer 做全自注意力计算，相似度评估极精准，适合精筛。
HyDE 表示对齐 Hypothetical Document Embeddings。先用大模型生成假设性答案，再向量化去检索真实文档，弥补问答表达鸿沟。
Triple (SPO) 图谱基本单元 三元组：主语 (Subject) - 谓语 (Predicate) - 宾语 (Object)，关系知识的最简表达形式。
Entity Resolution 实体消歧 在图构建中，将不同词汇（如“张三”、“张经理”、“老张”）识别、合并为同一个实体，决定了图谱质量。
Community Summary GraphRAG 核心 利用 Leiden 算法对大图自动进行聚类，并使用 LLM 生成社区总结报告，赋予模型进行 Global Reasoning 的能力。
Local / Global Search 检索路由 Local 围绕局部实体拓扑邻域精细查询；Global 读取聚类 Community Summary 归纳趋势与宏观概括。
Tool Retrieval (MCP) 能力检索 在执行链前将工具 JSON Schema 向量化并动态召回 Top-K 工具注入 LLM，解决能力膨胀下的上下文爆炸与幻觉。
