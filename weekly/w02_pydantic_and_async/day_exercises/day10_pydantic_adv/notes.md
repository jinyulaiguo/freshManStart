# 📅 Day 10 学习笔记：Pydantic 进阶与 Dataclass 对比

## 1. 运行时类型契约校验的本质区别

*   **问：为什么不能在需要强数据边界保障的场景下使用 Python 原生的 `dataclass`？**
*   **答：** 
    `dataclass` 的核心定位是**数据容器的魔术方法生成器**。它仅在**静态类型检查阶段**（如使用 `mypy`）起作用，在 Python 运行时完全不进行类型验证与数据拦截。
    而 `Pydantic` 则是在运行期构造函数内嵌了**数据校验拦截链**：
    1. **解析与强转 (Coercion)**：尝试将输入值转换为标注的期望类型（如将字符串 `"25"` 转换为整数 `25`）。
    2. **校验拦截 (Validation)**：若类型不符且无法强转，或违反自定义约束，立即抛出 `ValidationError`，阻止脏数据向系统内部流动。

| 特性 | `dataclass` (标准库) | `Pydantic BaseModel` (v2) |
| :--- | :--- | :--- |
| **运行时类型校验** | ❌ 默认无任何校验（默许通过） |  运行时强类型校验（校验失败即拦截） |
| **自动类型转换 (Coercion)** | ❌ 无转换（保持输入原样） |  自动转换（如 `"123"` $\rightarrow$ `123`, `"true"` $\rightarrow$ `True`） |
| **字段自定义验证** | ❌ 需在 `__post_init__` 中手写 |  支持 `@field_validator` (字段级精细校验) |
| **多字段联动验证** | ❌ 需在 `__post_init__` 中手写 |  支持 `@model_validator` (模型级联合校验) |
| **导出 JSON Schema** | ❌ 不原生支持 |  原生支持（`model_json_schema()`） |

## 2. 字段级校验器 (`field_validator`) 运行机制

*   **问：`field_validator` 的拦截顺序是怎样的？它在何时执行？**
*   **答：**
    在 Pydantic 中，字段级校验器默认是**在字段默认的类型强转和校验之后**执行的（即默认 `mode="after"`）。
    *   **方法特征**：校验器必须定义为类方法（使用 `@classmethod`），接收被校验的字段值。
    *   **数据流向**：输入数据 $\rightarrow$ Pydantic 内置解析（如强转为 `str`） $\rightarrow$ 传递给 `field_validator` $\rightarrow$ 返回校验/清洗后的值。如果未通过校验，必须抛出 `ValueError` 或 `AssertionError`，Pydantic 会将其包装成统一的 `ValidationError` 异常。

## 3. 模型级校验器 (`model_validator`) 运行机制

*   **问：什么时候应该使用 `model_validator`？它的 `mode` 参数有何讲究？**
*   **答：**
    当校验逻辑依赖于**多个字段之间的联合约束关系**时（例如：密码与确认密码是否一致、年龄与家长同意书是否关联），必须使用模型级校验器。
    
    `mode` 常见的两个参数值：
    1. **`mode="after"` (推荐)**：
       *   **执行时机**：在所有单个字段自己的类型转换和 `field_validator` 校验都通过后执行。
       *   **入参形式**：方法接收 `self`，此时属性已被绑定到实例上。
       *   **返回要求**：必须返回 `self`（或者相同类型的实例）。
    2. **`mode="before"`**：
       *   **执行时机**：在任何字段校验、类型转换甚至默认值填充之前执行。
       *   **入参形式**：接收一个原始的字典（`dict`）或其他输入数据结构。
       *   **适用场景**：需要在数据结构变形、清洗或字段重命名（Alias 映射）前进行全局预处理。

## 4. Agent 架构中的应用决策

*   **问：在构建大语言模型 Agent 框架时，如何权衡使用 `dataclass` 与 `Pydantic`？**
*   **答：**
    *   **选用 `dataclass` 的场景**：系统内部状态机流转数据（如 LangGraph 中的 `State` 容器）。因为状态在内部流转，类型已知且安全，使用 `dataclass` 可以避免不必要的校验开销，提升系统性能。
    *   **选用 `Pydantic` 的场景**：
        1. **外部边界输入输出**：LLM 吐出的 Tool 调用参数（必须通过 Pydantic 校验器拦截幻觉或格式错误字段）。
        2. **对外开放 API / HTTP 请求的 Payload**：作为防御性编程的第一道安全屏障。
