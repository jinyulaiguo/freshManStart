# 📅 Day 11 学习笔记：异步编程基础

## 1. 同步与异步的本质区别（阻塞 vs 非阻塞）

*   **问：在多核 CPU 普及的今天，为什么大模型 Agent 依然高度依赖单线程的异步编程，而不是单纯依靠多线程或多进程？**
*   **答：**
    这是由 **I/O 密集型任务**与**计算密集型任务**的硬件消耗本质决定的：
    1. **多线程/多进程的物理局限**：线程和进程的创建、销毁及上下文切换（Context Switch）由操作系统内核调度，开销极高。在面对成百上千个网络 I/O 连接时（例如大规模并发抓取网页或调度数十个 API 工具），线程会因等待网络响应而被挂起，大量闲置线程会迅速耗尽系统内存与内核调度资源。
    2. **同步阻塞（Blocking I/O）**：在同步模式下，当程序发起网络请求（如调用 LLM API），当前执行流必须原地等待网络数据传输完毕。在此期间，CPU 被强制剥夺控制权，程序无法响应任何其他任务，吞吐量极低。
    3. **异步非阻塞（Non-blocking I/O）**：异步编程利用单线程内的**协同式多任务调度**。当协程发起网络 I/O 请求时，它会主动将 CPU 的控制权让渡给事件循环（Event Loop），使 CPU 在等待网络响应的空闲期去处理其他就绪任务。网络数据返回后，事件循环重新激活该协程。这种机制以极低的内存开销实现了高并发。

| 维度 | 同步阻塞（Blocking） | 异步非阻塞（Non-blocking / `asyncio`） |
| :--- | :--- | :--- |
| **线程占用** | 每个 I/O 操作独占一个线程，线程等待时被阻塞 | 单个线程通过事件循环调度多个协程，等待时让出控制权 |
| **上下文切换开销** | 昂贵的内核态切换（CPU 寄存器、栈空间重载） | 极其轻量级的用户态协程切换（仅保存执行状态指针） |
| **系统资源消耗** | 高（受限于操作系统的线程数限制，通常数千个就会内存耗尽） | 极低（几十万个活跃协程可能仅消耗数十兆字节内存） |
| **典型应用场景** | 本地文件读写、纯 CPU 数值计算（如特征提取） | 大规模 API 轮询、并发工具调用、高并发 Web 接口 |

## 2. 协程声明与挂起：`async def` 与 `await`

*   **问：在 Python 中，`async def` 定义的函数和普通函数有什么区别？`await` 关键字在背后做了什么？**
*   **答：**
    1. **协程函数与协程对象**：
       使用 `async def` 声明的函数称为**协程函数 (Coroutine Function)**。当调用这个函数时，它**不会立即执行内部代码**，而是返回一个**协程对象 (Coroutine Object)**。
       ```python
       async def fetch_data():
           return "data"
       
       coro = fetch_data()  # 仅返回协程对象，不执行函数体
       ```
    2. **`await` 的挂起机制**：
       `await` 是一个表达式，其后必须紧跟一个**可等待对象 (Awaitable Object)**，如协程对象、`asyncio.Task` 或 `asyncio.Future`。
       *   **挂起点拦截**：当执行流遇到 `await` 时，协程会**暂停自身执行**并将控制权交回事件循环。
       *   **非阻塞等待**：事件循环在底层等待该 I/O 事件就绪，同时继续调度其他已就绪的协程。
       *   **恢复执行**：当被等待的协程执行完毕并返回结果时，事件循环会唤醒原协程，并从 `await` 表达式处继续向下执行。

## 3. 事件循环（Event Loop）与启动入口 `asyncio.run()`

*   **问：如何让协程跑起来？`asyncio.run()` 承担了什么角色？**
*   **答：**
    协程必须依附于**事件循环（Event Loop）**才能运行。事件循环是一个无限循环的内核，它持续监听系统 I/O 事件、定时器并调度协程任务队列。
    
    `asyncio.run(coro)` 是 Python 3.7+ 推荐的异步程序**顶层入口**，它的核心动作如下：
    1. **创建循环**：在当前线程中构建一个全新的事件循环实例。
    2. **注册主协程**：将传入的协程对象（通常是 `async def main()`）包装为 `Task` 并提交给事件循环。
    3. **运行并等待**：运行事件循环，直到主协程完全执行完毕并返回。
    4. **资源清理**：关闭事件循环，取消所有未完成的任务，释放异步资源。
    
    > [!IMPORTANT]
    > `asyncio.run()` 不允许在已经存在运行中事件循环的线程内重复调用。如果在交互式环境（如 Jupyter Notebook）或已有异步框架（如 FastAPI）中，直接 `await coro` 即可，不可再次调用 `asyncio.run()`。

## 4. 并发任务调度 `asyncio.gather`

*   **问：如何实现真正的并发调度？如果其中一个任务崩溃了，`asyncio.gather` 的默认行为是什么？**
*   **答：**
    如果仅串行使用 `await`，程序仍是顺序执行的，无法发挥并发优势：
    ```python
    # ❌ 错误示范：这仍然是串行的！总耗时 = delay_a + delay_b
    await mock_api_call("A", 2)
    await mock_api_call("B", 3)
    ```
    要实现并发，必须将多个协程提交给事件循环统一调度。`asyncio.gather(*aws, return_exceptions=False)` 提供了这一功能：
    1. **任务包装与注册**：`gather` 会自动将传入的协程包装为 `asyncio.Task` 并注册到事件循环中，使其**并发开始运行**。
    2. **执行时间特征**：总运行时间取决于**耗时最长的那一个可等待对象**，而非累加。
    3. **异常处理机制 (`return_exceptions`)**：
       *   `return_exceptions=False`（默认值）：若其中一个协程抛出异常，`gather` 会**立即中断并向外抛出该异常**。其他未完成的任务**不会被自动取消**，它们仍在后台继续运行，这可能引发资源悬空。
       *   `return_exceptions=True`：异常会被视作正常的返回结果，放入最终的返回列表里（类型为具体的 Exception 对象），不影响其他任务的收集。

## 5. Agent 架构中的并发 Tool 调度场景

*   **问：在多智能体（Multi-Agent）或 RAG 系统中，为什么并发任务调度至关重要？**
*   **答：**
    在生产环境中，Agent 在决策后可能会产出多个并行的子任务：
    1. **并行网页检索**：同时调用 Bing API 和 Google API 搜索 5 个不同的检索词。如果串行调用，每次请求耗时 1s，总计需要 5s；而使用 `asyncio.gather` 并发调度，总耗时仅为 ~1s。
    2. **多智能体并发协同**：当主 Agent 分发任务给“程序员 Agent”和“安全审计 Agent”时，这两个 Agent 应并发工作，而不是互相等待。
    3. **多模型评估与并票**：向 3 个不同的 LLM 实例（如 GPT-4, Claude 3.5, DeepSeek）并发发送相同的 prompt，收集它们的回答并进行多数投票（Majority Voting）。
