Week 7: 经典 RAG 知识体系与演进深度思考

本周的核心思想是通过“自底向上 (Bottom-Up)”的拼装哲学，构建一个解决企业扫描件/表格提取失真、字符与语义自适应分块、长文本注意力遗忘（Lost in the Middle）以及无匹配事实生成幻觉的高可靠经典 RAG 管道。

在文档准备阶段，我们的目标是将非结构化的原始介质转换为向量数据库能够完美索引的高保真“语义块”。这一过程经历了以下三个演进阶段：

第一步 (Day 48) 多模态解析与元数据富集

具体做了什么：利用 pdfplumber 提取 PDF 中的多列排版文本与表格，将其转为 Markdown 表格，并富集 doc_id 与 page 等文件级元数据。
演进关联：这是整个系统的水源。如果这里将表格解析成了碎字符，后续的所有分块与向量匹配都将彻底失效。

↓ 解决物理结构破坏后，进入字符级分层切分
第二步 (Day 43) 递归字符切片与 Overlap 平滑

具体做了什么：根据 ['\n\n', '\n', ' ', ''] 优先级递归寻找最自然的物理断句点，并回溯收集重叠片段。
演进关联：对于有明确语法缩进的代码文档（如 Python 函数），Day 43 能最大程度保持函数体结构的物理完整；但对于长篇的业务合规说明，它依然无法识别真正的话题突变边界。

↓ 解决字符切分无视语义的盲区，进入相似度自适应切分
第三步 (Day 44) 基于向量突变点的语义分块 (Semantic Chunking)

具体做了什么：将文档降级到句子级计算 Embedding 相似度，寻找相似度差值大于平均值加标准差的“语义突变点”进行物理切分。
演进关联：这是分块技术的终极演进。它让每个 Chunk 都成为了一个高度纯净、不夹杂冗余话题的“独立语义事实”，直接决定了向量库的精准召回上限。

当用户提出查询时，系统如何确保生成的回复既不包含模型自身的“幻觉”，又不会因为长上下文信息过载而遗漏关键事实？这三个模块相互支撑，共同解决该痛点：

召回与拦截 (Day 45) RAG 经典编排与空检索 Fallback

当向量库检索出的 Top-K 相似度全部低于阈值（如 0.6）时，说明参考知识库中根本不存在该事实。此时直接抛出 Null Fallback 拦截模型调用，防止模型在没有参考依据时瞎编。

↓ 若匹配成功，为了应对大模型处理长上下文的注意力衰减问题
信息排布优化 (Day 47) 黄金位置重排序 (Golden Position Reordering)

大模型在极长上下文的中间部分会发生严重的注意力流失（Lost in the Middle）。我们将最相似的几个 Chunk 交叉分布在 Prompt 上下文的最开头和最末尾，中等相关性的放在正中间，实现 U 型排布，迫使模型最先关注核心事实。

↓ 解决完注意力分布，还需要确保生成的每一句论点都有铁证支持
溯源证据链 (Day 46) 可信来源引用与脚注生成机制

在 System Prompt 中构建强契约限制，要求模型生成的每个事实都必须紧跟 [doc_id:page_number]。在 Python 侧流式捕捉并用正则还原原始 Chunk 对照。这使得生成的最终答案具备了可被人类二次核验的“铁证”，彻底解决了企业级落地时，RAG 系统被认为不可信的痛点。

在 Day 49，我们将上面所有的独立微引擎集成在一起，这不仅是代码的简单拼接，更是对自底向上（Bottom-Up）软件工程思想的绝佳实践：

“拼装积木墙”架构设计

  • 物理隔离：多格式解析器（Ingestor）、分块拆分器（Splitter）、重排处理器（Reorderer）和可信引用提取器（CitationParser）被封装在完全独立的文件和类中。
  • 单兵作战与单元测试：在将它们拼装成完整的 RAG 控制管道之前，每一个微引擎都已经通过了各自的单元测试（例如，确保分块器在 chunk_overlap >= chunk_size 时能主动报错拦截，而不是死锁；确保重排器输出符合 U 型位置契约）。这保障了在拼装时，底层的基石是 100% 坚固的。
  • 主入口无策略化：主运行控制程序（如 app.py 或 main.py）只负责协调实例的生命周期、数据流方向（Pipeline routing）以及提供命令行交互 REPL 界面，不承担任何算法逻辑的实现，使得系统具备极高的演进与维护效率。

Day 43
分块基础

固定大小 vs 递归字符分块

底层崩溃隐患 / 痛点
无视自然语义边界，固定字数切块会导致一个完整的代码段（如 Python 异步函数）或者核心学术概念被拦腰折断，高维表征的相关度大幅跌落。
工程解决方案 / 做了什么
使用递归字符分块，优先寻找最自然的段落/句子分隔符（\n\n, \n, 空格），尽量把语义聚拢。并在相邻块引入合理的 Overlap，平滑跨块边缘的信息丢失。
Agent 核心关联与价值
分块过大容易产生信息稀释且爆出 Context Window；分块过小会破坏上下文完整度。分块参数微调直接控制了后续向量匹配的上限。
🎯 过关验证标准
手写一个递归分块函数，传入长篇技术文档，验证在不同 chunk_size 和 chunk_overlap 设定下，生成的 Chunk 块中是否能完美保留完整的 Python 异步函数体，而不发生代码行被暴力砍断。

Day 43
分块策略（一）		 基于分隔符列表 ['\n\n', '\n', ' ', ''] 递归尝试切分长文本，利用 Overlap 控制邻接块信息平滑过渡。 防止固定大小分块在中间暴烈截断代码结构或句子，确保向量表征的数据连续性。 Fixed vs Recursive Splitter 脚本 + 异步函数切片保持度验证。
Day 44
分块策略（二）		 计算相邻单句的 Embedding 余弦相似度，利用统计学阈值（如均值加 1 倍标准差）寻找语义突变切分点。 传统分块仅做字符检测。语义分块基于话题转换自动切分，使每个 Chunk 都是纯净的“语义事实”。 SemanticTextSplitter 类 + 跨领域话题长文章（量子/中餐）切分分布打印。
Day 45
RAG 经典编排		 原生组合检索召回、提示词拼接生成；当相似度低于设定值（如0.6）时触发 Null Fallback。 在向量检索不到相关事实时，大模型会凭空产生强烈的幻觉并胡编乱造，需要物理拦截。 RAGPipeline 架构 + 低相似度 Query 拦截测试。
Day 46
可信引用脚注		 入库时富集 metadata，在 System Prompt 强规制输出引用 [doc_id:page]，正则抓取并还原原文。 企业级问答必须能证明每句话的出处，否则用户或审查人员根本无法信任生成的结论。 带 [doc_id:page] 标签的流式输出提取器 + 引用溯源原文高亮。
Day 47
黄金重排		 将 Top-K Chunk 列表中语义相似度最高的值排在首尾，中等强度的塞在中间。 大模型对超长 Prompt 的中间部分注意力分配极度下滑，导致中间的关键事实被彻底遗忘。 ReorderContextHandler + U 型重排规则位置打印验证。
Day 48
多格式解析		 基于 pdfplumber 提取 PDF 表格为 Markdown 语法并归一化，富集文件级别 metadata。 非结构化 PDF 或多栏文档提取易破损、表格数据结构崩坏导致大模型语义理解丧失。 MultiFormatDocIngestor + 混合表格 PDF 高保真 Markdown 提取。
Day 49
大项目集成		 自底向上将所有微引擎组件拼装为公司规章制度问答 Bot，通过 Qdrant 进行向量召回。 从多格式文件读取到向量数据库写入，再到重排、LLM 引用生成，打通复杂工业 RAG 系统闭环。 规章制度问答命令行 REPL + 完整单元测试 + 运行日志产物。
